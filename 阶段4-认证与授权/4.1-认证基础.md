## 1. 认证 vs 授权：别再搞混了

这是面试必考题，也是实际开发中最容易被混淆的概念：

```
认证（Authentication）  →  你是谁？  →  "请出示身份证"
授权（Authorization）   →  你能做什么？ →  "VIP 可以进，普通用户不行"
```

| | 认证 | 授权 |
|------|------|------|
| **英文** | Authentication | Authorization |
| **问什么** | 你是谁？ | 你能做什么？ |
| **常见方式** | 密码、JWT、OAuth2 | RBAC、ACL、权限装饰器 |
| **失败响应** | 401 Unauthorized | 403 Forbidden |
| **发生时机** | 先认证，再授权 | 认证通过后才检查 |

```
请求到达
    │
    ▼
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   认证        │ ──→ │   授权        │ ──→ │   业务逻辑    │
│  你是谁？     │ ✅  │  你能做吗？   │ ✅  │  执行操作     │
└──────────────┘     └──────────────┘     └──────────────┘
    │ 401                  │ 403
    ▼                      ▼
  未登录                 无权限
```

> 本章聚焦**认证**，第 4.3 章聚焦**授权**。

## 2. 密码处理：绝对不能明文存密码

### 2.1 为什么明文存密码是灾难？

```python
# ❌ 绝对不要这样做！
user = User(username="张三", password="12345678")  # 密码直接存数据库！
db.add(user)

# 数据库被拖库后：
# username | password
# -------- | --------
# 张三     | 12345678   ← 攻击者可以直接登录任何账号
```

### 2.2 哈希（Hash）——单项加密

```python
import hashlib

# 哈希的特点：
# 1. 同样的输入 → 同样的输出（可验证）
# 2. 无法从输出反推输入（单向）
# 3. 微小输入差异 → 完全不同的输出（雪崩效应）

password = "12345678"
hashed = hashlib.sha256(password.encode()).hexdigest()
print(hashed)  # ef797c8118f02dfb...  ← 存这个！

# 验证时：把用户输入的密码再哈希一次，和数据库比对
```

### 2.3 加盐（Salt）——防止彩虹表攻击

```python
import hashlib
import os

# ❌ 问题：相同密码 → 相同哈希值
# 用户A: 123456 → abc123
# 用户B: 123456 → abc123  ← 攻击者一看就知道两人密码相同！

# ✅ 加盐：每个用户一个随机盐值
def hash_password(password: str) -> tuple[str, str]:
    salt = os.urandom(16).hex()  # 随机盐
    hashed = hashlib.sha256((password + salt).encode()).hexdigest()
    return hashed, salt

def verify_password(password: str, hashed: str, salt: str) -> bool:
    return hashlib.sha256((password + salt).encode()).hexdigest() == hashed

# 用户A: 123456 + salt_a → hash_a
# 用户B: 123456 + salt_b → hash_b  ← 完全不同！
```

### 2.4 用 Passlib + bcrypt（实际项目的做法）

```bash
pip install passlib[bcrypt]
```

```python
from passlib.context import CryptContext

# 创建密码上下文
pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")

# 哈希密码
def hash_password(password: str) -> str:
    return pwd_context.hash(password)

# 验证密码
def verify_password(plain_password: str, hashed_password: str) -> bool:
    return pwd_context.verify(plain_password, hashed_password)

# ── 使用 ──
hashed = hash_password("my_secure_password")
print(hashed)  # $2b$12$LJ3m4ys3G...

# 验证
print(verify_password("my_secure_password", hashed))   # True
print(verify_password("wrong_password", hashed))       # False
```

> bcrypt 内置了盐值和迭代次数，是业界标准的密码哈希算法。

## 3. HTTP Basic Auth

最原始的 HTTP 认证方式，理解它的原理有助于理解更高级的方案。

### 工作原理

```
客户端                               服务器
   │                                   │
   ├── GET /admin ──────────────────→  │
   │                                   ├── 401 + WWW-Authenticate: Basic
   │ ←─────────────────────────────────┤
   │                                   │
   ├── GET /admin ──────────────────→  │
   │   Authorization: Basic base64(user:pass)
   │                                   ├── 验证成功 → 200
   │ ←─────────────────────────────────┤
```

### 在 FastAPI 中实现

```python
from fastapi import FastAPI, Depends, HTTPException, status
from fastapi.security import HTTPBasic, HTTPBasicCredentials
import secrets

app = FastAPI()
security = HTTPBasic()


def verify_basic_auth(credentials: HTTPBasicCredentials = Depends(security)):
    """验证 Basic Auth 凭证"""
    # ⚠️ 示例！实际应从数据库查用户并验证哈希
    correct_username = "admin"
    correct_password = "secret123"

    # 用 secrets.compare_digest 防止时序攻击
    is_correct_username = secrets.compare_digest(
        credentials.username, correct_username
    )
    is_correct_password = secrets.compare_digest(
        credentials.password, correct_password
    )

    if not (is_correct_username and is_correct_password):
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="用户名或密码错误",
            headers={"WWW-Authenticate": "Basic"},
        )
    return credentials.username


@app.get("/admin")
def admin_panel(username: str = Depends(verify_basic_auth)):
    return {"message": f"欢迎, {username}"}
```

### HTTP Basic Auth 的优缺点

| 优点 | 缺点 |
|------|------|
| 实现简单 | 密码用 Base64 编码（≈明文）传输 |
| HTTP 标准，所有客户端支持 | 必须搭配 HTTPS |
| 适合内部工具、脚本 | 无法设置过期时间 |
| 无需 Cookie/Session | 不支持登出、权限分级 |

> **结论：** 仅适合内部工具或测试。生产环境用户认证请用 JWT（4.2 章）或 OAuth2。

## 4. Session 认证（传统 Web 应用）

### 工作原理

```
登录流程：
  用户提交用户名+密码
       │
       ▼
  服务器验证密码
       │
       ▼
  服务器创建 Session（存入 Redis/数据库/内存）
  返回 Session ID（通常放在 Cookie 里）
       │
       ▼
后续请求：
  浏览器自动携带 Cookie
       │
       ▼
  服务器查 Session → 获取用户信息
```

### 简易实现（用于理解原理）

```python
import uuid
from datetime import datetime, timedelta
from fastapi import FastAPI, Depends, HTTPException, Cookie, Response

app = FastAPI()

# ═══════ 模拟 Session 存储（实际项目用 Redis） ═══════
sessions: dict[str, dict] = {}  # session_id → {"username": ..., "expires_at": ...}


def create_session(username: str) -> str:
    """创建 Session，返回 Session ID"""
    session_id = uuid.uuid4().hex
    sessions[session_id] = {
        "username": username,
        "expires_at": datetime.utcnow() + timedelta(hours=2),
    }
    return session_id


def get_session(session_id: str) -> dict | None:
    """从 Session ID 获取用户信息"""
    session = sessions.get(session_id)
    if not session:
        return None
    if datetime.utcnow() > session["expires_at"]:
        del sessions[session_id]
        return None
    return session


# ═══════ 模拟用户数据库 ═══════
FAKE_USERS = {
    "admin": {"username": "admin", "password": "hashed_secret123"},
    "user": {"username": "user", "password": "hashed_password"},
}


# ═══════ 路由 ═══════

@app.post("/login")
def login(username: str, password: str, response: Response):
    """登录——创建 Session，返回 Set-Cookie"""
    user = FAKE_USERS.get(username)
    if not user or user["password"] != "hashed_" + password:
        raise HTTPException(status_code=401, detail="用户名或密码错误")

    # 创建 Session
    session_id = create_session(username)

    # 通过 Set-Cookie 把 Session ID 发给浏览器
    response.set_cookie(
        key="session_id",
        value=session_id,
        httponly=True,   # JavaScript 无法读取（防 XSS）
        max_age=7200,    # 2 小时过期
        samesite="lax",  # 防 CSRF
    )
    return {"message": f"登录成功, {username}"}


@app.post("/logout")
def logout(response: Response, session_id: str = Cookie(default=None)):
    """登出——删除服务端 Session + 清除客户端 Cookie"""
    if session_id and session_id in sessions:
        del sessions[session_id]
    response.delete_cookie("session_id")
    return {"message": "已登出"}


@app.get("/me")
def get_current_user(session_id: str = Cookie(default=None)):
    """获取当前登录用户"""
    session = get_session(session_id) if session_id else None
    if not session:
        raise HTTPException(status_code=401, detail="未登录或 Session 已过期")
    return {"username": session["username"]}


@app.get("/protected")
def protected_route(session_id: str = Cookie(default=None)):
    """受保护的路由——需要登录才能访问"""
    session = get_session(session_id) if session_id else None
    if not session:
        raise HTTPException(status_code=401, detail="请先登录")
    return {"data": "这是敏感信息", "user": session["username"]}
```

### Session vs Token

| | Session | JWT Token |
|------|---------|-----------|
| **状态** | 有状态（服务端存储） | 无状态（自包含） |
| **存储位置** | 服务端（Redis/DB） | 客户端（localStorage/Cookie） |
| **扩展性** | 需要共享 Session 存储 | 天然支持分布式 |
| **适用场景** | 传统 Web 应用（服务端渲染） | API / SPA / 微服务 |
| **登出控制** | 服务端直接删除 | 需要黑名单或等过期 |

> FastAPI 是 API 框架，**推荐使用 JWT Token 而非 Session**。第 4.2 章详细介绍。

## 5. OAuth2 协议基础

OAuth2 是当今最主流的授权协议，FastAPI 内置了完整支持。

### OAuth2 的四种角色

```
┌─────────────┐     ┌──────────────┐
│  Resource   │     │  Authorization │
│  Owner      │     │  Server       │
│  (用户)      │     │  (认证服务器)    │
└──────┬──────┘     └──────┬───────┘
       │                   │
       │  ① 授权           │
       │◄──────────────────┤
       │                   │
       │  ② 返回 Token      │
       │──────────────────►│
                            │
                    ┌───────▼────────┐
                    │  Resource      │
                    │  Server        │
                    │  (API 服务器)    │
                    └────────────────┘
```

| 角色 | 说明 | 举例 |
|------|------|------|
| **Resource Owner** | 资源拥有者（用户） | 张三 |
| **Client** | 客户端应用 | 第三方 App |
| **Authorization Server** | 认证服务器 | GitHub OAuth |
| **Resource Server** | 资源服务器 | FastAPI API |

### 最常用的 OAuth2 流程：Password Flow

```
POST /token
Content-Type: application/x-www-form-urlencoded

grant_type=password&username=zhang&password=12345678&scope=read

      ↓

Response:
{
    "access_token": "eyJhbGciOi...",
    "token_type": "bearer",
    "expires_in": 3600
}
```

> FastAPI 的 `OAuth2PasswordBearer` + `OAuth2PasswordRequestForm` 实现了这个流程。第 4.2 章完整实现。

## 6. 将认证封装为 FastAPI 依赖

不管用哪种认证方式（Basic Auth / Session / JWT），最终都封装为 FastAPI 依赖：

```python
# 依赖：获取当前用户（所有受保护路由共用）
def get_current_user(token: str = Depends(oauth2_scheme)) -> User:
    """从 Token 中解析出当前登录用户"""
    payload = decode_token(token)
    user = db.get(User, payload["sub"])
    if not user:
        raise HTTPException(status_code=401, detail="用户不存在")
    return user

# 端点：只关心业务逻辑，不关心里面怎么解析 token 的
@app.get("/me")
def read_current_user(current_user: User = Depends(get_current_user)):
    return {"username": current_user.username, "email": current_user.email}
```

**依赖注入的好处：**
1. 所有端点共享同一份认证逻辑
2. 修改认证方式只需改依赖函数，端点不变
3. 测试时可以用 `app.dependency_overrides` 替换
