# 6.1 后台任务

> 请求处理完就立刻返回给用户——耗时操作（发邮件、写日志、推送通知）丢给后台慢慢做，不拖慢响应速度。

---

## 1. 为什么需要后台任务？

```python
@app.post("/register")
def register(data: UserCreate, db: Session = Depends(get_db)):
    # 1. 创建用户
    user = User(username=data.username, ...)
    db.add(user)
    db.commit()

    # 2. 发送欢迎邮件（耗时 3 秒！）
    send_welcome_email(data.email)       # ← 用户要干等 3 秒才拿到响应
    return {"id": user.id}
```

| 不用后台任务 | 用后台任务 |
|------------|-----------|
| 注册接口响应时间 = 50ms（写入DB）+ 3s（发邮件）= 3.05s | 注册接口响应时间 = 50ms |
| 用户看到白屏/转圈 3 秒 | 用户秒收响应 |
| 发邮件失败 → 注册也失败 | 发邮件失败 → 不影响注册（可重试） |

> 💡 **核心原则：响应时间只包含"用户必须等待"的操作。发邮件、写日志、推通知——用户不需要等这些。**

## 2. BackgroundTasks 基础用法

FastAPI 内置了 `BackgroundTasks`，通过依赖注入使用：

```python
from fastapi import APIRouter, BackgroundTasks

router = APIRouter()

def send_welcome_email(email: str, username: str):
    """模拟发送欢迎邮件（耗时操作）"""
    import time
    time.sleep(3)  # 模拟：实际是调 SMTP 或邮件 API
    print(f"✅ 欢迎邮件已发送给 {email}")
    # 实际代码：
    # smtp.send(
    #     to=email,
    #     subject=f"欢迎 {username}",
    #     body="感谢注册！"
    # )

@router.post("/register")
def register(
    background_tasks: BackgroundTasks,  # ← 注入 BackgroundTasks
    db: Session = Depends(get_db),
):
    # 创建用户...
    user = User(...)
    db.add(user)
    db.commit()

    # 把耗时任务"丢"给后台 → 请求立即返回
    background_tasks.add_task(send_welcome_email, user.email, user.username)
    #                                          ↑ 函数名       ↑ 参数（按顺序）

    return {"id": user.id}  # ← 用户马上收到了这个响应
```

### 工作流程

```
请求到达 POST /register
    │
    ├─ 1. 执行业务逻辑（创建用户）     ← 在请求-响应周期内
    ├─ 2. background_tasks.add_task() ← 只是"注册"任务，不执行
    ├─ 3. 返回响应给用户                ← 用户秒收！
    │
    └─ 4. 响应发送完毕后               ← 后台执行
        └─ send_welcome_email() 执行   ← 用户已经收到响应了
```

## 3. 多个后台任务

可以添加多个任务，它们会按添加顺序依次执行：

```python
@router.post("/register")
def register(background_tasks: BackgroundTasks, db: Session = Depends(get_db)):
    # 创建用户...
    user = User(...)
    db.add(user)
    db.commit()

    # 添加多个后台任务
    background_tasks.add_task(send_welcome_email, user.email, user.username)
    background_tasks.add_task(log_registration, user.id, user.username)
    background_tasks.add_task(send_notification_to_admin, user.username)

    return {"id": user.id}

# 执行顺序: send_welcome_email → log_registration → send_notification_to_admin
```

> ⚠️ 后台任务是**串行**执行的，不是并行的。如果需要并行，考虑 Celery 等任务队列。

## 4. 实际场景

### 4.1 发送邮件

```python
import smtplib
from email.mime.text import MIMEText

def send_email(to_email: str, subject: str, body: str):
    """后台发送邮件"""
    msg = MIMEText(body, "html", "utf-8")
    msg["Subject"] = subject
    msg["From"] = "noreply@myapp.com"
    msg["To"] = to_email

    with smtplib.SMTP("smtp.example.com", 587) as server:
        server.starttls()
        server.login("user", "password")
        server.send_message(msg)

@router.post("/users")
def create_user(data: UserCreate, background_tasks: BackgroundTasks, db: Session = Depends(get_db)):
    user = User(...)
    db.add(user)
    db.commit()

    background_tasks.add_task(
        send_email,
        user.email,
        "欢迎注册",
        f"<h1>你好 {user.username}！</h1><p>感谢注册。</p>"
    )
    return {"id": user.id}
```

### 4.2 写入审计日志

```python
import json

def write_audit_log(action: str, user_id: int, detail: dict):
    """后台写审计日志"""
    log_entry = {
        "action": action,
        "user_id": user_id,
        "detail": detail,
        "timestamp": datetime.utcnow().isoformat(),
    }
    with open("audit.log", "a") as f:
        f.write(json.dumps(log_entry, ensure_ascii=False) + "\n")

@router.delete("/users/{user_id}")
def delete_user(
    user_id: int,
    current_user: User = Depends(require_admin),
    background_tasks: BackgroundTasks,
    db: Session = Depends(get_db),
):
    user = db.get(User, user_id)
    db.delete(user)
    db.commit()

    # 记录审计日志——不拖慢响应
    background_tasks.add_task(
        write_audit_log,
        "user_deleted",
        current_user.id,
        {"target_user_id": user_id, "target_username": user.username},
    )
    return {"message": "已删除"}
```

### 4.3 生成缩略图（图片处理）

```python
from PIL import Image

def generate_thumbnail(image_path: str, size=(200, 200)):
    """后台生成缩略图"""
    img = Image.open(image_path)
    img.thumbnail(size)
    thumb_path = image_path.replace(".", "_thumb.")
    img.save(thumb_path)
    return thumb_path

@router.post("/upload/image")
async def upload_image(
    file: UploadFile,
    background_tasks: BackgroundTasks,
):
    # 保存原图
    content = await file.read()
    image_path = f"uploads/{file.filename}"
    with open(image_path, "wb") as f:
        f.write(content)

    # 后台生成缩略图
    background_tasks.add_task(generate_thumbnail, image_path)

    return {"message": "图片已上传，缩略图稍后生成", "path": image_path}
```

## 5. BackgroundTasks 的局限性

| 局限 | 说明 | 后果 |
|------|------|------|
| **无持久化** | 任务存在内存中 | 服务器重启 → 任务丢失 |
| **无重试** | 执行失败就失败了 | 邮件没发出去 → 永远不会补发 |
| **无进度** | 无法查询任务状态 | 不知道任务是否执行完了 |
| **串行执行** | 任务按顺序排队 | 前一个慢任务堵住后一个 |

```
BackgroundTasks 适用：
  ✅ 写日志
  ✅ 发送不需要保证到达的通知
  ✅ 更新非关键数据（统计计数等）

BackgroundTasks 不适用：
  ❌ 支付回调（需要重试 + 状态追踪）
  ❌ 批量导出报表（需要进度反馈）
  ❌ 需要定时调度的任务
  ❌ 高并发大量任务
```

## 6. 升级方案：从 BackgroundTasks 到 Celery

当后台任务变多变复杂时，需要专业的任务队列：

| | BackgroundTasks | Celery |
|------|---------|--------|
| **持久化** | ❌ 内存 | ✅ Redis / RabbitMQ |
| **重试** | ❌ 手动 | ✅ 内置自动重试 |
| **进度查询** | ❌ 不支持 | ✅ 可通过状态查询 |
| **分布式** | ❌ 单进程 | ✅ 多 Worker 并行 |
| **定时任务** | ❌ 不支持 | ✅ Celery Beat |
| **复杂度** | 零配置 | 需要 Redis + Worker |

### Celery 快速预览

```python
# tasks.py
from celery import Celery

celery_app = Celery("myapp", broker="redis://localhost:6379/0")

@celery_app.task(bind=True, max_retries=3, default_retry_delay=60)
def send_welcome_email(self, email: str, username: str):
    """Celery 任务 —— 失败自动重试 3 次，间隔 60 秒"""
    try:
        smtp.send(to=email, subject=f"欢迎 {username}")
    except Exception as exc:
        raise self.retry(exc=exc)  # 失败时自动重试

# 在端点中触发
@router.post("/register")
def register(data: UserCreate, db: Session = Depends(get_db)):
    user = User(...)
    db.add(user)
    db.commit()

    # Celery 任务 —— 持久化到 Redis，失败自动重试
    send_welcome_email.delay(user.email, user.username)
    #                  ↑ delay = 异步发送到任务队列

    return {"id": user.id}
```

## 本章小结

```
BackgroundTasks 决策树：

  这个操作能等吗？
    │
    ├─ 能等 → 直接写在请求里（如数据校验）
    │
    └─ 不能等（耗时 > 100ms）
        │
        ├─ 不需要重试/持久化 → BackgroundTasks ✅
        │   （发邮件、写日志、生成缩略图）
        │
        └─ 需要重试/持久化/进度 → Celery / ARQ
            （支付回调、报表导出、批量处理）
```

| 要点 | 说明 |
|------|------|
| 注入方式 | `background_tasks: BackgroundTasks` 作为函数参数 |
| 添加任务 | `background_tasks.add_task(func, arg1, arg2)` |
| 执行时机 | 响应发送给用户**之后** |
| 执行顺序 | FIFO 串行 |
| 适用场景 | 邮件、日志、推送、缩略图、统计 |
