# 6.4 API 文档与版本化

> 自动生成的 Swagger 很好，但你可以让它更好——学会定制 OpenAPI 文档、管理多个 API 版本、让前端和第三方的对接体验拉满。

---

## 1. 自定义 OpenAPI 文档

FastAPI 默认生成 OpenAPI 3.0 规范的文档。你可以通过 `app` 参数和 `openapi()` 方法定制：

### 1.1 修改文档基本信息

```python
app = FastAPI(
    title="我的 API 服务",
    description="""
    这是一个电商平台的 API 服务。

    ## 功能模块
    - **用户管理**：注册、登录、信息修改
    - **商品管理**：CRUD、分类、搜索
    - **订单管理**：创建、支付、物流追踪

    ## 认证
    使用 JWT Bearer Token。在 Authorize 按钮中输入 Token。
    """,
    version="2.1.0",
    terms_of_service="https://example.com/terms",
    contact={
        "name": "API 支持团队",
        "url": "https://example.com/support",
        "email": "api@example.com",
    },
    license_info={
        "name": "MIT",
        "url": "https://opensource.org/licenses/MIT",
    },
    openapi_tags=[  # ← 自定义分组标签（控制 Swagger UI 中的分组顺序）
        {"name": "用户", "description": "用户注册、登录、信息管理"},
        {"name": "商品", "description": "商品的 CRUD 和搜索"},
        {"name": "订单", "description": "订单生命周期管理"},
        {"name": "管理", "description": "管理员专用接口"},
    ],
    openapi_url="/api/v1/openapi.json",     # 自定义 OpenAPI JSON 路径
    docs_url="/api/v1/docs",                 # 自定义 Swagger UI 路径
    redoc_url="/api/v1/redoc",               # 自定义 ReDoc 路径
)
```

### 1.2 Swagger UI 中的效果

```
┌─────────────────────────────┐
│  我的 API 服务  [v2.1.0]    │
│  这是一个电商平台...          │
├─────────────────────────────┤
│  用户                        │  ← openapi_tags 定义的分组
│  POST /register   注册       │
│  POST /login      登录       │
│  GET  /users/me   获取信息   │
├─────────────────────────────┤
│  商品                        │
│  GET  /products    列表      │
│  POST /products    创建      │
├─────────────────────────────┤
│  授权按钮 → 🔓 Authorize     │
└─────────────────────────────┘
```

### 1.3 用 `router` 的 tags 参数分组

```python
# routers/users.py
router = APIRouter(tags=["用户"])       # ← 路由层指定 tags

@router.post("/register")
def register(...): ...

# routers/products.py
router = APIRouter(tags=["商品"])

@router.get("/products")
def list_products(...): ...
```

> 💡 **最佳实践：** 每个 `APIRouter` 设一个 `tags`，在 `app` 的 `openapi_tags` 中统一排序和描述。

## 2. 定制 OpenAPI Schema

### 2.1 修改全局 OpenAPI Schema

```python
from fastapi.openapi.utils import get_openapi

def custom_openapi():
    """自定义 OpenAPI Schema"""
    if app.openapi_schema:
        return app.openapi_schema

    openapi_schema = get_openapi(
        title="我的 API",
        version="1.0.0",
        description="自定义 Schema 描述",
        routes=app.routes,
    )

    # ── 添加自定义字段 ──
    openapi_schema["info"]["x-logo"] = {
        "url": "https://example.com/logo.png"
    }
    openapi_schema["info"]["x-api-id"] = "com.example.api"

    # ── 在全局级别添加安全方案 ──
    openapi_schema["components"]["securitySchemes"] = {
        "BearerAuth": {
            "type": "http",
            "scheme": "bearer",
            "bearerFormat": "JWT",
            "description": "输入 JWT Token（不含 Bearer 前缀）",
        },
        "APIKeyHeader": {
            "type": "apiKey",
            "in": "header",
            "name": "X-API-Key",
        },
    }

    app.openapi_schema = openapi_schema
    return app.openapi_schema


app.openapi = custom_openapi  # ← 替换默认的 openapi 生成函数
```

### 2.2 给端点加更多文档信息

```python
from fastapi import APIRouter, Path, Query

router = APIRouter()

@router.post(
    "/products",
    response_model=ProductResponse,
    status_code=201,
    summary="创建商品",                          # ← Swagger 列表中的简短标题
    description="创建一个新商品。需要 editor 或 admin 角色。",  # ← 展开后的详细说明
    response_description="创建的商品对象",         # ← 响应说明
    tags=["商品"],
    deprecated=False,                            # True = 标记为"已废弃"
    include_in_schema=True,                      # False = 在文档中隐藏
)
def create_product(
    data: ProductCreate,
    db: Session = Depends(get_db),
):
    """创建新商品"""  # ← docstring 会作为 description（如果没写 description 参数）
    ...
```

### 2.3 隐藏内部端点

```python
# 方式 1：不在文档中显示
@app.get("/internal/health", include_in_schema=False)
def internal_health():
    """内部健康检查，不暴露到文档"""
    return {"status": "ok"}

# 方式 2：控制 docs_url（无认证的用户看不到文档）
app = FastAPI(docs_url="/docs" if settings.ENVIRONMENT != "production" else None)
```

## 3. API 版本化策略

### 三种主流版本化方式

```
策略 1: URL 前缀        /api/v1/users, /api/v2/users
策略 2: 请求头           Accept: application/vnd.myapi.v2+json
策略 3: 查询参数         /users?version=2
```

| 策略 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| **URL 前缀** | 直观、容易理解、好测试 | URL 不"永久" | ⭐⭐⭐⭐⭐ 最推荐 |
| **请求头** | URL 不变 | 文档难展示、不易调试 | ⭐⭐⭐ |
| **查询参数** | 简单 | 不规范、容易被忽略 | ⭐⭐ |

### 3.1 URL 前缀版本化（推荐）

#### 项目结构

```
app/
├── api/
│   ├── v1/
│   │   ├── __init__.py
│   │   ├── router.py         # 注册所有 v1 路由
│   │   └── endpoints/
│   │       ├── users.py
│   │       └── products.py
│   └── v2/
│       ├── __init__.py
│       ├── router.py         # 注册所有 v2 路由
│       └── endpoints/
│           ├── users.py      # 新版用户接口
│           └── products.py
└── main.py
```

#### 代码实现

```python
# main.py
from fastapi import FastAPI
from app.api.v1.router import router as v1_router
from app.api.v2.router import router as v2_router

app = FastAPI()

# 分别挂载两个版本
app.include_router(v1_router, prefix="/api/v1", tags=["V1"])
app.include_router(v2_router, prefix="/api/v2", tags=["V2"])
```

```python
# app/api/v1/router.py
from fastapi import APIRouter
from app.api.v1.endpoints import users, products

router = APIRouter()
router.include_router(users.router, prefix="/users")
router.include_router(products.router, prefix="/products")

# ═══ 最终路径 ═══
# /api/v1/users/register      → v1.endpoints.users.register
# /api/v1/users/login         → v1.endpoints.users.login
# /api/v1/products            → v1.endpoints.products.list_products
```

```python
# app/api/v2/router.py
from fastapi import APIRouter
from app.api.v2.endpoints import users, products

router = APIRouter()
router.include_router(users.router, prefix="/users")
router.include_router(products.router, prefix="/products")

# ═══ 最终路径 ═══
# /api/v2/users/register      → v2.endpoints.users.register（可能返回不同格式）
# /api/v2/products            → v2.endpoints.products.list_products（可能多了字段）
```

#### v1 vs v2 的差异示例

```python
# v1/endpoints/users.py — 旧版
@router.post("/register", response_model=UserResponseV1)
def register(data: UserCreateV1, ...):
    """
    V1 注册：只需 username + password
    """
    ...

# v2/endpoints/users.py — 新版
@router.post("/register", response_model=UserResponseV2)
def register(data: UserCreateV2, ...):
    """
    V2 注册：增加了手机号、验证码
    """
    ...

# schema 变化
class UserCreateV1(BaseModel):
    username: str
    password: str

class UserCreateV2(BaseModel):
    username: str
    password: str
    phone: str                        # ← V2 新增
    verification_code: str            # ← V2 新增
```

## 4. API 废弃（Deprecation）

### 4.1 标记废弃

```python
@router.get("/api/v1/users/old-list", deprecated=True)
def old_list_users():
    """
    旧版用户列表 —— 已废弃，请迁移到 /api/v2/users

    ⚠️ 此接口将于 2026-12-31 移除
    """
    ...

# Swagger UI 会显示 ⚠️ Deprecated 标记
```

### 4.2 Sunset 通知头部

```python
from fastapi import Response

@router.get("/api/v1/products")
def list_products_v1(response: Response):
    """
    已废弃。请使用 /api/v2/products。
    """
    # 通过响应头告知客户端
    response.headers["Deprecation"] = "true"
    response.headers["Sunset"] = "Sat, 31 Dec 2026 23:59:59 GMT"
    response.headers["Link"] = '</api/v2/products>; rel="successor-version"'

    # 返回旧版数据格式...
```

### 4.3 渐进式废弃流程

```
第 1 阶段 — 发布新版本：
  ✅ 创建 /api/v2/products
  ✅ 旧接口 /api/v1/products 正常使用

第 2 阶段 — 通知废弃（1 个月后）：
  ⚠️ /api/v1/products 标记 deprecated=True
  ⚠️ 响应头添加 Deprecation + Sunset

第 3 阶段 — 灰度迁移（2 个月后）：
  📊 /api/v1/products 返回警告日志，统计仍在调用的客户端
  📧 通知剩余用户迁移

第 4 阶段 — 移除（3 个月后）：
  ❌ /api/v1/products 返回 HTTP 410 Gone
```

## 5. 安全性：隐藏/保护文档

### 5.1 生产环境隐藏文档

```python
import os

ENV = os.getenv("ENV", "development")

app = FastAPI(
    docs_url=None if ENV == "production" else "/docs",
    redoc_url=None if ENV == "production" else "/redoc",
    openapi_url=None if ENV == "production" else "/openapi.json",
)
```

### 5.2 文档加认证保护

```python
from fastapi.openapi.docs import get_swagger_ui_html

@app.get("/docs", include_in_schema=False)
async def protected_docs(user: User = Depends(get_current_user)):
    """只有登录用户可以看文档"""
    if user.role != "admin":
        raise HTTPException(403, "只有管理员可以查看 API 文档")
    return get_swagger_ui_html(
        openapi_url="/openapi.json",
        title="API 文档",
    )
```

## 6. 增强 Swagger UI：用 `servers` 多环境切换

```python
app = FastAPI(
    servers=[
        {"url": "http://127.0.0.1:8000", "description": "本地开发环境"},
        {"url": "https://staging-api.example.com", "description": "测试环境"},
        {"url": "https://api.example.com", "description": "生产环境"},
    ],
)
```

Swagger UI 中会出现一个下拉菜单，让你切换环境——一键对不同的服务器发请求！

## 本章小结

```
API 文档与版本化全景：

  文档定制                版本管理              废弃策略
  ┌──────────┐         ┌──────────┐         ┌──────────┐
  │ Swagger  │         │ /api/v1/ │ ──to──> │ /api/v2/ │
  │ ReDoc    │         │ /api/v2/ │         │ deprecate│
  │ 自定义   │         │ Header   │         │ Sunset   │
  │ 隐藏/认证│         │ Query    │         │ 410 Gone │
  └──────────┘         └──────────┘         └──────────┘
```

| 要点 | 最佳实践 |
|------|---------|
| **文档分组** | `openapi_tags` 定义分组，`APIRouter(tags=[...])` 分配 |
| **版本化** | URL 前缀 `/api/v1/` → `/api/v2/`，结构清晰 |
| **废弃流程** | deprecated → Sunset 头 → 410 Gone，给用户迁移时间 |
| **生产安全** | 生产环境隐藏或加认证保护 docs_url |
| **多环境** | `servers` 参数配置多个环境 URL |

## 实战练习

1. 为你当前的 API 添加 `openapi_tags`，让 Swagger 文档按功能模块分组
2. 创建 v1 和 v2 两套 `/users` 接口（v2 增加新字段），用 URL 前缀区分
3. 给旧的 v1 接口添加 Deprecation 和 Sunset 响应头
4. 配置 `servers` 参数，在 Swagger UI 中实现多环境切换
