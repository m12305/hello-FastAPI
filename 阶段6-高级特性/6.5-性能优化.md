## 1. 先定位，后优化——性能分析

**不要凭感觉优化！先用数据说话。**

### 1.1 最简单的方式：中间件计时

```python
import time
from fastapi import Request

@app.middleware("http")
async def add_process_time_header(request: Request, call_next):
    """给每个响应加上 X-Process-Time 头（毫秒）"""
    start = time.time()
    response = await call_next(request)
    process_time = (time.time() - start) * 1000
    response.headers["X-Process-Time"] = f"{process_time:.2f}ms"
    return response

# 响应头示例：
# X-Process-Time: 152.34ms
```

### 1.2 测量数据库查询时间

```python
from sqlalchemy import event
from sqlalchemy.engine import Engine
import time

@event.listens_for(Engine, "before_cursor_execute")
def before_cursor_execute(conn, cursor, statement, parameters, context, *args):
    conn.info.setdefault("query_start_time", []).append(time.time())

@event.listens_for(Engine, "after_cursor_execute")
def after_cursor_execute(conn, cursor, statement, parameters, context, *args):
    elapsed = time.time() - conn.info["query_start_time"].pop()
    if elapsed > 0.1:  # 只记录超过 100ms 的慢查询
        print(f"🐌 慢查询 ({elapsed:.2f}s): {statement[:200]}")
```

## 2. 数据库查询优化

### 2.1 索引：最立竿见影的优化

```python
# ❌ 没有索引
# SELECT * FROM users WHERE email = 'test@example.com'  → 全表扫描

# ✅ 加了索引
class User(Base):
    __tablename__ = "users"

    id: Mapped[int] = mapped_column(primary_key=True)
    email: Mapped[str] = mapped_column(
        String(100), unique=True, index=True  # ← index=True
    )
    username: Mapped[str] = mapped_column(
        String(50), unique=True, index=True
    )
    # 复合索引（常用组合查询）
    __table_args__ = (
        Index("ix_users_role_is_active", "role", "is_active"),  # ← 复合索引
    )

# SELECT * FROM users WHERE role='editor' AND is_active=1  → 命中复合索引
```

**何时加索引：**

```
加索引：
  ✅ WHERE 条件频繁使用的列
  ✅ JOIN 外键列（SQLAlchemy 自动为 ForeignKey 加索引）
  ✅ ORDER BY 排序的列
  ✅ 唯一约束的列

不加索引：
  ❌ 经常变动的列（每次更新都要重建索引）
  ❌ 高重复率的列（如 is_active、gender——区分度低）
  ❌ 小表（< 1000 行，全表扫描也很快）
```

### 2.2 只查需要的列（SELECT 字段）

```python
# ❌ 查全部列（SELECT *）
users = db.query(User).all()

# ✅ 只查需要的列
users = db.query(User.id, User.username, User.email).all()
# → SELECT id, username, email FROM users

# ✅ 用 load_only 限制加载
from sqlalchemy.orm import load_only

users = db.execute(
    select(User).options(load_only(User.id, User.username))
).scalars().all()
```

### 2.3 解决 N+1 查询（用 joinedload / selectinload）

这是最重要的是数据库优化技巧：

```python
# ❌ N+1 问题：查 100 篇文章 = 1 + 100 = 101 条 SQL！
posts = db.query(Post).all()          # 第 1 条 SQL
for post in posts:
    print(post.author.username)      # 每篇文章触发第 N 条 SQL！

# ✅ 用 joinedload：1 条 SQL JOIN 解决
from sqlalchemy.orm import joinedload, selectinload

posts = db.execute(
    select(Post).options(joinedload(Post.author))  # ← JOIN users
).scalars().unique().all()
# → SELECT * FROM posts JOIN users ON posts.user_id = users.id

# ✅ 大数据量用 selectinload：2 条 SQL 解决
posts = db.execute(
    select(Post).options(selectinload(Post.author))  # ← IN 查询
).scalars().all()
# → 第 1 条: SELECT * FROM posts
# → 第 2 条: SELECT * FROM users WHERE id IN (1,2,3,...,100)
```

| 策略 | SQL 条数 | 适用 |
|------|---------|------|
| Lazy loading | 1 + N 条 ❌ | 几乎不用 |
| **joinedload** | 1 条 | 数据少、一对一、一对少 |
| **selectinload** | 2 条 | 数据多、一对多、多对多 |

### 2.4 分页（减少传输量）

```python
@app.get("/users")
def list_users(page: int = Query(1, ge=1), size: int = Query(20, le=100)):
    offset = (page - 1) * size
    users = db.query(User).offset(offset).limit(size).all()
    total = db.query(User).count()
    return {
        "data": users,
        "page": page,
        "size": size,
        "total": total,
        "pages": (total + size - 1) // size,
    }

# 游标分页（大数据量更优）：
@app.get("/users")
def list_users(cursor: int | None = None, size: int = 20):
    query = select(User).order_by(User.id).limit(size + 1)
    if cursor:
        query = query.where(User.id > cursor)
    users = db.execute(query).scalars().all()

    has_more = len(users) > size
    if has_more:
        users = users[:size]

    return {
        "data": users,
        "next_cursor": users[-1].id if has_more else None,
    }
```

### 2.5 连接池调优

```python
from sqlalchemy import create_engine

engine = create_engine(
    "mysql+pymysql://user:pass@host/db",
```

## 3. 异步 vs 同步——战略选择

### 3.1 什么时候用 async？

```
async def 适合的场景：
  ✅ 调用外部 API（httpx.AsyncClient）
  ✅ 数据库 I/O（asyncpg, asyncmy）
  ✅ Redis（redis.asyncio）
  ✅ 文件读写（aiofiles）
  ✅ WebSocket

def 就够了：
  ✅ CPU 密集型计算（Math、图片处理等）
  ✅ 纯内存操作
  ✅ 使用同步库（SQLAlchemy 同步 ORM、requests 等）
```

### 3.2 注意不要混用！

```python
# ❌ 灾难：在 async def 中调用同步阻塞函数
import time

@app.get("/slow")
async def slow_endpoint():
    time.sleep(5)              # ← 阻塞了整个事件循环！5 秒内不处理任何请求！
    return {"status": "ok"}

# ✅ 正确：在 async def 中使用异步库
import asyncio

@app.get("/fast")
async def fast_endpoint():
    await asyncio.sleep(5)     # ← 不阻塞，5 秒内可以处理其他请求
    return {"status": "ok"}

# ✅ 如果必须用同步阻塞函数，用线程池
from concurrent.futures import ThreadPoolExecutor
import asyncio

@app.get("/workaround")
async def workaround():
    loop = asyncio.get_event_loop()
    result = await loop.run_in_executor(None, blocking_function)
    return {"result": result}
```

### 3.3 async def 路由 + def 路由共存

```python
# FastAPI 会自动在单独的线程池中运行 def 路由
# 所以简单 CRUD 用 def 也没问题

@app.get("/users")           # def 路由 → 在线程池中执行（不阻塞事件循环）
def list_users(db: Session = Depends(get_db)):
    return db.query(User).all()

@app.get("/external-data")    # async def 路由 → 在事件循环中执行
async def get_external():
    async with httpx.AsyncClient() as client:
        r = await client.get("https://api.example.com/data")
    return r.json()
```

## 4. 响应缓存策略

### 4.1 使用 Cache-Control 头

```python
from fastapi import Response

@app.get("/news")
def get_news(response: Response):
    # 告诉浏览器/代理：这个数据 5 分钟内不变
    response.headers["Cache-Control"] = "public, max-age=300"
    return {"news": "..."}

@app.get("/user/profile")
def get_profile(response: Response):
    # 私有数据——只允许浏览器缓存，不允许 CDN 缓存
    response.headers["Cache-Control"] = "private, max-age=60"
    return {"username": "..."}
```

| 指令 | 含义 |
|------|------|
| `public` | 允许 CDN 和中间代理缓存 |
| `private` | 只允许浏览器缓存 |
| `max-age=300` | 缓存 300 秒 |
| `no-cache` | 每次使用前先验证（用 ETag） |
| `no-store` | 完全不缓存 |

### 4.2 ETag——减少传输

```python
import hashlib
import json

@app.get("/products")
def list_products(request: Request, response: Response):
    products = db.query(Product).all()
    data = [{"id": p.id, "name": p.name, "price": p.price} for p in products]

    # 计算 ETag（数据的哈希值）
    etag = hashlib.md5(json.dumps(data, sort_keys=True).encode()).hexdigest()

    # 如果客户端的 ETag 和服务器一致 → 返回 304 Not Modified
    if request.headers.get("If-None-Match") == etag:
        response.status_code = 304
        return Response(status_code=304)

    response.headers["ETag"] = etag
    response.headers["Cache-Control"] = "public, max-age=60"
    return data
```

## 6. 性能优化检查清单

```
□ 数据库：
  □ 常用查询列加了索引吗？
  □ N+1 查询用 joinedload/selectinload 解决了吗？
  □ 列表接口做了分页吗？
  □ 连接池配置合理吗（pool_size, pool_recycle）？

□ 查询：
  □ 只 SELECT 需要的列吗？
  □ 用了 Bulk Insert 批量写入吗？
  □ 大事务拆小了吗？

□ 响应：
  □ 启用了 GZip 中间件吗？
  □ 静态数据加了 Cache-Control 吗？
  □ 列表接口返回字段做了过滤吗？

□ 架构：
  □ 阻塞函数用了 run_in_executor 吗？
  □ 耗时任务用了 BackgroundTasks 或 Celery 吗？
  □ 接口有超时控制吗？
```

## 本章小结

```
性能优化优先级（从最有效到最复杂）：

  1. 加索引          → 效果：10x~100x，成本：低
  2. 解决 N+1 查询   → 效果：5x~50x，成本：低
  3. 加分页          → 效果：数据量越大越明显，成本：低
  4. 连���池调优      → 效果：2x~5x，成本：中
  5. 加缓存          → 效果：10x~100x，成本：中
  6. 加 GZip         → 效果：带宽减少 80%，成本：极低
  7. 异步/多Worker   → 效果：吞吐量提升，成本：高
```

| 维度 | 最佳实践 |
|------|---------|
| **数据库** | 索引 → joinedload → 分页 → 连接池 |
| **异步** | async def + 异步驱动（asyncmy/asyncpg） |
| **缓存** | Cache-Control + ETag + Redis |
| **压缩** | GZipMiddleware（几乎零成本） |
| **压测** | wrk 快速验证，locust 做复杂场景 |
