# 2.3 依赖注入（Dependency Injection）

> ⚡ **这是 FastAPI 最重要的概念**——没有之一。依赖注入让你用声明式的方式管理代码复用、认证、数据库连接，它贯穿了 FastAPI 的整个设计哲学。

---

## 1. 什么是依赖注入？

### 1.1 用一个比喻理解

想象你在经营一家餐厅。每位客人点菜时，服务员都需要：
1. 检查客人是否预定了座位（认证）
2. 给客人倒水、递菜单（准备上下文）
3. 记录点菜内容（日志）

如果没有依赖注入，每个服务员（端点函数）都要亲自跑一遍所有步骤。代码到处都是重复的"验证预定 → 倒水 → 递菜单"。

**依赖注入就像是给餐厅配了一个迎宾员**——客人进门时，迎宾员统一完成预定检查、倒水、递菜单，然后把准备好的客人信息交给服务员。服务员只专注于"点菜"这件事。

在 FastAPI 中，依赖注入让**每个端点只关注自己的业务逻辑**，而把认证、数据库连接、权限检查等通用逻辑抽出来复用。

### 1.2 代码对比：没有 vs 有依赖注入

```python
# ❌ 没有依赖注入：认证逻辑在每个端点中重复

@app.get("/users/me")
async def get_current_user(authorization: str = Header(...)):
    # 😫 解析 Token...（和下面一模一样）
    token = authorization.replace("Bearer ", "")
    try:
        payload = jwt.decode(token, SECRET_KEY, algorithms=[ALGORITHM])
    except JWTError:
        raise HTTPException(status_code=401)
    user = await db.get_user(payload["sub"])
    if not user:
        raise HTTPException(status_code=401)
    return user


@app.get("/users/me/posts")
async def get_my_posts(authorization: str = Header(...)):
    # 😫 又要重复一遍！每个端点都要复制粘贴这段代码
    token = authorization.replace("Bearer ", "")
    ...
```

**核心痛点**：同样的认证逻辑在 10 个端点上重复了 10 次。哪天认证逻辑要改（比如新增 Token 刷新机制），10 个端点全要改——这非常容易出错。

```python
# ✅ 用依赖注入：认证逻辑写一次，到处复用

from fastapi import Depends

async def get_current_user(authorization: str = Header(...)):
    """依赖函数：认证逻辑集中在这里，写一次就够了"""
    token = authorization.replace("Bearer ", "")
    try:
        payload = jwt.decode(token, SECRET_KEY, algorithms=[ALGORITHM])
    except JWTError:
        raise HTTPException(status_code=401)
    user = await db.get_user(payload["sub"])
    if not user:
        raise HTTPException(status_code=401)
    return user


@app.get("/users/me")
async def read_current_user(
    current_user = Depends(get_current_user),  # ← 一行代码完成认证！
):
    return current_user


@app.get("/users/me/posts")
async def read_my_posts(
    current_user = Depends(get_current_user),  # ← 复用同一依赖，零重复
):
    posts = await db.get_posts_by_user(current_user.id)
    return posts
```

**三行代码解决所有问题**：定义依赖 → 注入 → 复用。修改认证逻辑只需改 `get_current_user` 一个函数。

## 2. `Depends()` 基本语法

来看一个更简单的例子。假设很多端点都需要分页参数（`page`、`page_size` 和计算 `offset`），我们可以把这段逻辑封装成一个依赖函数：

```python
from fastapi import FastAPI, Depends

app = FastAPI()

# ─── 依赖函数：它就是一个普通的 Python 函数 ───
def common_parameters(
    q: str | None = None,
    page: int = 1,
    page_size: int = 20,
):
    """把分页逻辑集中在这里"""
    return {
        "q": q,
        "page": page,
        "page_size": page_size,
        "offset": (page - 1) * page_size,  # SQL 查询用的偏移量
    }


# ─── 在端点中注入依赖 ───
@app.get("/items/")
async def read_items(
    params: dict = Depends(common_parameters),  # ← 注入！
):
    """params 就是 common_parameters 的返回值"""
    return params


@app.get("/users/")
async def read_users(
    params: dict = Depends(common_parameters),  # ← 同一个依赖复用到不同端点
):
    return {"users": [...], **params}
```

### Depends() 做了什么？—— 自动分析参数

`Depends()` 最精妙的地方在于：它会**自动分析依赖函数的参数**，然后像分析路径操作函数一样，自动从请求中提取对应的数据。

```
浏览器请求: GET /items/?q=laptop&page=2&page_size=10
                │
                ▼
FastAPI 发现 Depends(common_parameters)
                │
                ▼
分析 common_parameters 的签名:
    ├── q: str | None = None       → 从查询参数中取 q=laptop
    ├── page: int = 1              → 从查询参数中取 page=2
    └── page_size: int = 20        → 从查询参数中取 page_size=10
                │
                ▼
调用 common_parameters(q="laptop", page=2, page_size=10)
                │
                ▼
返回值: {"q": "laptop", "page": 2, "page_size": 10, "offset": 10}
                │
                ▼
赋值给 read_items 的 params 参数
```

**核心原理**：`Depends()` 中的函数参数可以是 `Query()`、`Path()`、`Header()`、`Cookie()` 甚至另一个 `Depends()`——FastAPI 会递归解析所有参数，自动从请求的各个位置提取数据。

## 3. 依赖的四种形态

FastAPI 的依赖系统非常灵活——依赖不一定是函数，它可以是任何**可调用对象**。根据使用场景的不同，依赖主要有四种形态：

| 形态 | 什么时候用 | 特点 |
|------|-----------|------|
| **函数依赖** | 最常用，日常首选 | 简单直接，同步/异步都支持 |
| **类 + `__call__`** | 需要带配置的依赖 | 通过构造函数传参，创建不同配置的依赖实例 |
| **类本身（无 `__call__`）** | 简单的参数组合 | FastAPI 自动实例化，省代码 |
| **`yield` 依赖** | 需要清理资源的依赖 | 数据库连接、文件句柄等"用完要关"的场景 |

### 3.1 函数依赖（最常用——日常首选）

这就是最简单的玩法——写一个普通的 Python 函数，用 `Depends()` 调用它。

```python
# 同步函数
def get_db():
    db = SessionLocal()
    try:
        yield db      # yield → 带清理的依赖（见第 5 节）
    finally:
        db.close()

# 异步函数（数据库查询、外部 API 调用等 I/O 操作推荐用这个）
async def get_current_user(token: str = Depends(oauth2_scheme)):
    user = await verify_token(token)
    return user
```

### 3.2 可调用对象（类 + `__call__`）—— 带配置的依赖

有时候同一个依赖逻辑需要**不同的配置**。比如普通用户每页最多看 100 条，但公开接口只能看 10 条（防止爬虫）。把配置参数放到 `__init__` 中，就能创建不同配置的依赖实例。

```python
class Pagination:
    def __init__(self, max_page_size: int = 100):
        """配置在初始化时传入，不同实例有不同的上限"""
        self.max_page_size = max_page_size

    def __call__(
        self,
        page: int = 1,
        page_size: int = 20,
    ) -> dict:
        """每次请求调用时执行"""
        if page_size > self.max_page_size:
            raise HTTPException(
                status_code=400,
                detail=f"page_size 不能超过 {self.max_page_size}",
            )
        return {
            "page": page,
            "page_size": page_size,
            "offset": (page - 1) * page_size,
        }


# 同一个类，两个不同配置的实例，应用到不同端点
default_pagination = Pagination(max_page_size=100)   # 内部接口：大分页
small_pagination = Pagination(max_page_size=10)       # 公开接口：小分页


@app.get("/items/")
async def read_items(p: dict = Depends(default_pagination)):
    """普通用户，最多每页 100 条"""
    return p


@app.get("/items/public/")
async def read_public_items(p: dict = Depends(small_pagination)):
    """公开接口，最多每页 10 条（防止爬虫和滥用）"""
    return p
```

> 这种模式的精髓：**同一个类的不同实例 → 不同的依赖行为**。逻辑代码只写一份（在 `__call__` 中），配置参数在实例化时区分。

### 3.3 类本身作为依赖（无 `__call__`）—— 最简洁的写法

如果不需要额外的配置参数，可以直接用类作为依赖。FastAPI 会自动调用 `__init__` 来实例化。

```python
class CommonQueryParams:
    def __init__(
        self,
        q: str | None = None,
        page: int = 1,
        page_size: int = 20,
    ):
        """__init__ 的参数自动从请求中提取"""
        self.q = q
        self.page = page
        self.page_size = page_size


@app.get("/items/")
async def read_items(
    # 完整写法：Depends(CommonQueryParams)  → FastAPI 自动实例化
    commons: CommonQueryParams = Depends(CommonQueryParams),
):
    return {"q": commons.q, "page": commons.page}


@app.get("/users/")
async def read_users(commons: CommonQueryParams = Depends()):
    """快捷写法：省略类名。类型标注告诉 FastAPI 这是 CommonQueryParams"""
    return {"q": commons.q, "page": commons.page}
```

> 快捷写法 `Depends()` 的原理：FastAPI 看参数类型标注是 `CommonQueryParams`，自动推导出应该实例化这个类。

### 3.4 `yield` 依赖（带清理的依赖）—— 最实用的形态

有些资源"用完必须还"——数据库连接需要关闭、文件需要保存后释放、锁需要解锁。`yield` 依赖就是为了自动管理这些资源的**生命周期**。

可以这样理解：`yield` 把函数分成两半，`yield` 之前是"开门"，`yield` 之后是"关门"。门在请求开始时打开，在请求结束时自动关闭——无论请求成功还是失败。

```python
class DatabaseSession:
    def __init__(self):
        self.id = id(self)
        print(f"🔗 Session {self.id} opened")

    def close(self):
        print(f"🔒 Session {self.id} closed")

    def query(self, sql: str):
        return f"Result of: {sql}"


async def get_db():
    """yield 依赖——请求处理完自动关闭数据库会话"""
    db = DatabaseSession()    # ← 第 1 步：请求到达时，创建会话
    try:
        yield db               # ← 第 2 步：将会话传给端点函数
    finally:
        db.close()             # ← 第 3 步：响应返回后，关闭会话（一定执行）


@app.get("/data")
async def read_data(db: DatabaseSession = Depends(get_db)):
    result = db.query("SELECT * FROM items")
    return {"data": result}
```

时间线：
```
请求到达 → 创建 DatabaseSession → yield db → 端点执行 → 响应返回 → finally 关闭
         └── yield 之前 ──┘              └── 端点 ──┘  └── yield 之后 ──┘
```

> 这就是后续章节中数据库会话管理的核心模式。后续学 SQLAlchemy 时，你会反复看到这种写法。

## 4. 依赖嵌套（Dependency Chains）

依赖可以依赖另一个依赖——形成依赖链：

```python
from fastapi import FastAPI, Depends, Header, HTTPException

app = FastAPI()


# ─── 第一层：提取 API Key ───
async def get_api_key(x_api_key: str = Header(alias="X-API-Key")):
    """从请求头提取 API Key"""
    return x_api_key


# ─── 第二层：用 API Key 查找用户 ───
async def get_current_user(
    api_key: str = Depends(get_api_key),  # ← 依赖上一层的返回值
):
    """用 API Key 查找对应的用户"""
    # 模拟数据库查询
    fake_users = {
        "key-admin-123": {"id": 1, "name": "Admin", "role": "admin"},
        "key-user-456":  {"id": 2, "name": "Alice", "role": "user"},
    }
    user = fake_users.get(api_key)
    if not user:
        raise HTTPException(status_code=401, detail="Invalid API Key")
    return user


# ─── 第三层：权限校验 ───
async def require_admin(
    current_user: dict = Depends(get_current_user),  # ← 依赖上一层的返回值
):
    """确保当前用户是管理员"""
    if current_user["role"] != "admin":
        raise HTTPException(
            status_code=403,
            detail="仅管理员可执行此操作",
        )
    return current_user


# ─── 端点使用 ───
@app.get("/admin/stats")
async def admin_stats(
    admin: dict = Depends(require_admin),  # ← 最终依赖
):
    """仅管理员可访问的数据"""
    return {
        "message": "管理员面板",
        "admin": admin["name"],
        "stats": {"users": 1250, "orders": 890},
    }
```

依赖图：

```
get_api_key       (提取 API Key)
    ↓
get_current_user  (API Key → 用户)
    ↓
require_admin     (用户 → 权限校验)
    ↓
admin_stats       (端点)
```

> FastAPI 会自动构建依赖图，**只调用一次**——即使同一个依赖被多个地方引用，每个请求周期内也只执行一次（缓存机制）。

## 5. `yield` 依赖详解

`yield` 依赖是最常用也最容易搞混的依赖形态。它主要用于那些**需要在使用完毕后清理资源**的场景——数据库连接、文件操作、网络请求等。

### 5.1 基础：生命周期管理

```python
async def get_db_session():
    """数据库会话——最经典的 yield 依赖"""
    print("→ 打开数据库连接")
    session = create_session()
    try:
        yield session           # ← 将 session 返回给端点使用
    finally:
        print("→ 关闭数据库连接")
        session.close()         # ← 无论成功还是异常，finally 保证执行


@app.get("/items/")
async def read_items(db=Depends(get_db_session)):
    items = db.execute("SELECT * FROM items")
    return items
    # 函数返回后，finally 块自动执行——你不需要手动关连接
```

### 5.2 yield 中的异常处理：两个不同的错误域

一个常见的困惑：`yield` 依赖中的 `except` 能捕获端点函数中抛出的异常吗？**不能。** yield 之前的代码和端点函数处于不同的错误处理域。

```python
async def get_db():
    session = create_session()
    try:
        yield session
    except Exception:
        # ⚠️ 这里只能捕获 yield 语句本身的异常，
        #    不能捕获端点函数中的异常！
        #    端点函数的异常由 FastAPI 的全局异常处理器处理。
        print("yield 本身出错（极少发生）")
        raise
    finally:
        session.close()  # finally 块始终执行，不受异常影响
```

> **记住**：`finally` 是你的朋友——它保证资源一定会被释放。`except` 在 yield 依赖中很少使用，因为端点函数中的异常不会传播到这里。

### 5.3 多个 `yield` 依赖的执行顺序

当多个 `yield` 依赖嵌套时，它们的清理顺序遵循**栈规则（后进先出，LIFO）**——后注册的依赖先清理：

```python
async def dep_a():
    print("A: 进入")
    try:
        yield "A"
    finally:
        print("A: 退出")


async def dep_b():
    print("B: 进入")
    try:
        yield "B"
    finally:
        print("B: 退出")


@app.get("/test")
async def test(
    a=Depends(dep_a),
    b=Depends(dep_b),
):
    print("端点执行中")
    return {"a": a, "b": b}

# 实际输出顺序:
# A: 进入        ← 先注册的先执行（进入）
# B: 进入
# 端点执行中
# B: 退出        ← 后注册的先执行（退出）—— 栈模式
# A: 退出
```

> 这个顺序很重要！比如 `dep_a` 是数据库连接，`dep_b` 依赖 `dep_a` 的数据。那么 `dep_b` 必须先清理（因为它依赖 `dep_a`），然后才能安全关闭 `dep_a`。

## 6. 依赖的缓存与作用域

### 6.1 默认行为：请求级缓存（重要！）

FastAPI 的依赖有一个关键的默认行为：**同一个请求中，同一个依赖只执行一次**。这是一种优化措施——如果两个不同的依赖都引用了同一个上游依赖（比如 `get_current_user`），那 `get_current_user` 只被调用一次，结果被缓存复用。

```python
async def expensive_dependency():
    print("💸 昂贵的计算（执行中）...")
    return {"data": "very expensive"}


@app.get("/b")
async def endpoint_b(
    dep1=Depends(expensive_dependency),
    dep2=Depends(expensive_dependency),  # ← 不会重新计算，直接复用上面的结果
):
    # dep1 is dep2  → True（同一个对象引用）
    # "💸 昂贵的计算..." 只打印一次
    return {"dep1": dep1, "dep2": dep2}
```

> 这个缓存行为让嵌套依赖的高效执行成为可能——深层依赖不会在每个引用点都被重复计算。

### 6.2 `use_cache=False` 禁用缓存

但有时你**希望**每次调用都重新执行。比如生成随机 ID、获取当前时间戳。此时用 `use_cache=False`：

```python
import random

def generate_request_id():
    """每次调用生成不同的 ID"""
    return f"req-{random.randint(1000, 9999)}"


@app.get("/ids")
async def get_ids(
    id1: str = Depends(generate_request_id, use_cache=False),
    id2: str = Depends(generate_request_id, use_cache=False),
):
    # id1 和 id2 会是不同的值（因为每次调用都重新执行）
    return {"id1": id1, "id2": id2}
```

## 7. 实用场景大全

下面的场景会贯穿你后续的全部学习——你会在阶段 3（数据库）、阶段 4（认证）、阶段 7（架构）中反复看到这些模式。

### 场景 1：请求日志记录

用 yield 依赖记录每个请求的耗时——yield 之前记录开始时间，yield 之后计算耗时并输出日志。这是所有 API 都应该有的基础能力。

```python
import time
import logging

logger = logging.getLogger("api")


async def log_request(
    request: Request,
    x_request_id: str | None = Header(default=None, alias="X-Request-ID"),
):
    start = time.time()
    logger.info(f"[{x_request_id}] {request.method} {request.url.path}")

    yield  # 等待端点执行...

    duration = time.time() - start
    logger.info(f"[{x_request_id}] 完成, 耗时 {duration:.3f}s")


@app.get("/items/")
async def read_items(_log=Depends(log_request)):
    # 日志自动记录，端点代码完全不需要关心
    return [{"id": 1}]
```

### 场景 2：用户认证（最常用——记住这个模式）

这是 FastAPI 依赖注入最典型的应用。`get_current_user` 从请求头提取 Token、解码、查数据库，然后把用户对象返回。所有需要登录的端点只需加一行 `Depends(get_current_user)`。

```python
from fastapi import Depends, HTTPException, status
from fastapi.security import OAuth2PasswordBearer

oauth2_scheme = OAuth2PasswordBearer(tokenUrl="/auth/login")


async def get_current_user(
    token: str = Depends(oauth2_scheme),  # 从 Authorization 头提取 Bearer Token
    db=Depends(get_db),                   # 依赖数据库会话
):
    """验证 JWT，返回当前用户对象——这是认证体系的核心依赖"""
    credentials_exception = HTTPException(
        status_code=status.HTTP_401_UNAUTHORIZED,
        detail="认证失败",
        headers={"WWW-Authenticate": "Bearer"},
    )

    try:
        payload = jwt.decode(token, SECRET_KEY, algorithms=[ALGORITHM])
        user_id: int = payload.get("sub")
        if user_id is None:
            raise credentials_exception
    except JWTError:
        raise credentials_exception

    user = await db.get_user(user_id)
    if user is None:
        raise credentials_exception

    return user


# 所有需要认证的端点，一行 Depends 搞定
@app.get("/users/me")
async def read_me(current_user=Depends(get_current_user)):
    return current_user
```

### 场景 3：分页参数

把分页逻辑封装成一个类依赖——所有列表接口都自动获得 `page`、`page_size`、`offset`。不需要在每个端点里手动计算。

```python
class PaginationParams:
    def __init__(
        self,
        page: int = Query(default=1, ge=1, description="页码"),
        page_size: int = Query(
            default=20,
            ge=1,
            le=100,
            alias="page-size",  # URL 中用 page-size，Python 中用 page_size
            description="每页条数",
        ),
    ):
        self.page = page
        self.page_size = page_size

    @property
    def offset(self) -> int:
        """SQL 查询用的偏移量——自动计算，不用手动写公式"""
        return (self.page - 1) * self.page_size

    @property
    def limit(self) -> int:
        return self.page_size


@app.get("/items/")
async def list_items(paging: PaginationParams = Depends()):
    items = await db.fetch_all(offset=paging.offset, limit=paging.limit)
    return {"page": paging.page, "items": items}
```

### 场景 4：权限校验——函数返回函数的"工厂模式"

这是一个进阶技巧。`require_role(role)` 本身不是一个依赖，它是一个**生成依赖的函数**（工厂函数）。调用它返回一个新的依赖函数，这个新函数已经内置了所需的角色。这样就实现了"一份代码，N 种角色"的效果。

```python
from enum import Enum


class Role(str, Enum):
    admin = "admin"
    editor = "editor"
    viewer = "viewer"


def require_role(role: Role):
    """工厂函数：传入角色，返回该角色的校验依赖"""
    async def role_checker(current_user=Depends(get_current_user)):
        if current_user.role != role:
            raise HTTPException(
                status_code=403,
                detail=f"需要 {role} 权限，当前角色为 {current_user.role}",
            )
        return current_user
    return role_checker  # ← 返回一个依赖函数


@app.get("/admin/dashboard")
async def admin_dashboard(admin=Depends(require_role(Role.admin))):
    return {"message": "管理员专属"}


@app.get("/editor/posts")
async def editor_posts(editor=Depends(require_role(Role.editor))):
    return {"message": "编辑者专属"}


# 如果将来需要新角色，只需 require_role(Role.new_role)，不需要改任何现有代码
```

### 场景 5：数据库会话管理（阶段 3 会深入）

这是你会在阶段 3 反复使用的模式——每个请求获取一个独立的数据库连接，请求结束时自动关闭。

```python
from sqlalchemy.orm import Session
from db import SessionLocal


def get_db():
    """每个请求获取独立的数据库会话——yield 依赖最经典的用法"""
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()  # 响应返回后自动关闭，不管你端点里有没有异常


@app.get("/users/")
async def list_users(db: Session = Depends(get_db)):
    return db.query(User).all()


@app.post("/users/")
async def create_user(user_data: UserCreate, db: Session = Depends(get_db)):
    user = User(**user_data.model_dump())
    db.add(user)
    db.commit()
    db.refresh(user)
    return user
```

## 8. 依赖作为路径装饰器参数

依赖不仅可以在函数参数中使用，还可以直接加到装饰器上：

```python
@app.get("/items/", dependencies=[Depends(verify_api_key)])
async def read_items():
    """verify_api_key 的返回值不需要，但校验必须通过"""
    return [{"id": 1}]


@app.get("/users/", dependencies=[Depends(verify_token)])
async def read_users():
    return [{"name": "Alice"}]


# 对整个路由器生效
from fastapi import APIRouter

router = APIRouter(
    prefix="/admin",
    dependencies=[Depends(require_admin)],  # ← 所有路由都需要管理员权限
)

@router.get("/stats")
async def stats():
    return {"users": 100}

@router.get("/config")
async def config():
    return {"theme": "dark"}
```

对比：

| 方式 | 何时使用 |
|------|---------|
| `param = Depends(fn)` | 需要依赖的**返回值** |
| `dependencies=[Depends(fn)]` | 只需要依赖的**副作用**（校验、日志），不需要返回值 |

## 9. 依赖注入 vs 中间件

| | 依赖注入 | 中间件 |
|------|---------|--------|
| 作用范围 | 选定端点或路由器 | 所有请求 |
| 粒度 | 精细（可指定到单个端点） | 全局 |
| 获取返回值 | ✅ 可以 | ❌ 无法直接传值 |
| 请求体访问 | ❌ 不能（依赖在请求体解析前执行） | ✅ 可以 |
| 典型用途 | 认证、分页、数据库会话 | 日志、CORS、请求计时 |

```python
# 依赖注入：精确控制哪些端点需要认证
@app.get("/public")
async def public(): ...

@app.get("/private")
async def private(user=Depends(get_current_user)): ...


# 中间件：对所有请求生效，无法选择
@app.middleware("http")
async def log_all_requests(request, call_next): ...
```
