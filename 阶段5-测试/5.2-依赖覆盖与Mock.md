# 5.2 依赖覆盖与 Mock

> 测试不应该真的发短信、扣钱、调第三方支付——用 FastAPI 的依赖覆盖机制优雅地替换外部依赖，让测试又快又稳又不要钱。

---

## 1. 问题：测试中的"外部依赖噩梦"

假设你有一个发送验证码的接口：

```python
# routers/users.py
from fastapi import APIRouter, Depends
import httpx  # 调用第三方短信 API

router = APIRouter()

def send_sms(phone: str, code: str):
    """调用第三方短信平台发送验证码——每次调用都花钱！"""
    response = httpx.post(
        "https://sms-api.example.com/send",
        json={"phone": phone, "content": f"验证码：{code}"},
        headers={"Authorization": "Bearer sk-xxx"},
    )
    return response.json()

@router.post("/send-code")
def send_verification_code(phone: str, sms=Depends(send_sms)):
    import random
    code = str(random.randint(100000, 999999))
    result = sms(phone, code)
    return {"message": "验证码已发送", "result": result}
```

直接测试会遇到的问题：

| 问题 | 后果 |
|------|------|
| 💰 **真的花钱** | 每跑一次测试发一条短信，跑 100 次 = 100 条短信费 |
| 🐌 **速度慢** | HTTP 调第三方 API 耗时 200-500ms，100 个测试就要等几十秒 |
| 🌐 **依赖网络** | 没网就不能跑测试，第三方挂了你的 CI 也挂了 |
| 🎲 **不可控** | 第三方返回什么你控制不了，无法测试"短信发送失败"的降级逻辑 |

**解决方案：Mock（模拟）——用假的换掉真的。**

## 2. FastAPI 的依赖覆盖机制

FastAPI 提供了一个极其优雅的方案：`app.dependency_overrides`。

```python
from myapp.main import app
from myapp.dependencies import get_db, send_sms

# ── 原理 ──
# app.dependency_overrides 是一个字典：
#   键 = 原始依赖函数
#   值 = 替换函数

app.dependency_overrides[send_sms] = fake_send_sms
#                              ↑                ↑
#                          "凡是调用 send_sms 的地方"
#                                         "改成调用 fake_send_sms"
```

### 2.1 最简单的例子

```python
# 原始依赖
def get_current_time():
    import time
    return time.time()  # 每次调用返回不同值 → 很难测试！

@app.get("/time")
def current_time(now: float = Depends(get_current_time)):
    return {"now": now}
```

测试时覆盖它：

```python
from fastapi.testclient import TestClient

def fake_get_current_time():
    return 1700000000.0  # 固定返回值

def test_current_time():
    app.dependency_overrides[get_current_time] = fake_get_current_time

    client = TestClient(app)
    response = client.get("/time")
    assert response.json() == {"now": 1700000000.0}  # 可预测！
```

### 2.2 覆盖数据库依赖（最常用）

这是测试中最常见的模式——用测试数据库替换生产数据库：

```python
# conftest.py
import pytest
from fastapi.testclient import TestClient
from sqlalchemy import create_engine
from sqlalchemy.orm import sessionmaker

from database import Base, get_db
from main import app

@pytest.fixture
def client():
    """用 SQLite 内存库覆盖真实数据库"""
    engine = create_engine("sqlite:///:memory:", connect_args={"check_same_thread": False})
    TestingSession = sessionmaker(bind=engine)

    Base.metadata.create_all(bind=engine)

    def override_get_db():
        db = TestingSession()
        try:
            yield db
        finally:
            db.close()

    app.dependency_overrides[get_db] = override_get_db  # ← 关键：替换！

    with TestClient(app) as c:
        yield c

    app.dependency_overrides.clear()  # ← 重要：清理！
    Base.metadata.drop_all(bind=engine)
```

> ⚠️ **一定要在测试结束后 `clear()` 覆盖！** 否则可能影响后续测试。

## 3. Mock 外部服务

### 3.1 用 `unittest.mock` 模拟函数

Python 自带 `unittest.mock`，不需要额外安装：

```python
from unittest.mock import MagicMock, patch

# ── MagicMock：万能替身 ──
fake_sms = MagicMock()
fake_sms.return_value = {"status": "ok", "message_id": "mocked-12345"}

# 调用 fake_sms 的任何参数都返回固定值
result = fake_sms("13800138000", "123456")
print(result)  # {"status": "ok", "message_id": "mocked-12345"}
```

结合依赖覆盖：

```python
# test_sms.py
from unittest.mock import MagicMock
from fastapi.testclient import TestClient
from main import app
from routers.users import send_sms  # 原始依赖函数

def test_send_code():
    # 创建 Mock
    mock_sms = MagicMock()
    mock_sms.return_value = {"status": "ok"}   # 不花一分钱！

    # 覆盖
    app.dependency_overrides[send_sms] = mock_sms

    client = TestClient(app)
    response = client.post("/send-code", params={"phone": "13800138000"})

    assert response.status_code == 200
    # 验证 send_sms 确实被调用了
    mock_sms.assert_called_once()
    # 验证传入的参数
    assert "13800138000" in str(mock_sms.call_args)

    app.dependency_overrides.clear()
```

### 3.2 模拟不同的返回值

测试"短信发送失败"的情况——在真实环境你很难让短信平台真的失败，但在测试中轻而易举：

```python
def test_send_code_failure():
    mock_sms = MagicMock()
    mock_sms.side_effect = Exception("短信平台连接超时")
    #         ↑ 不返回正常值，而是抛异常！

    app.dependency_overrides[send_sms] = mock_sms

    client = TestClient(app)
    response = client.post("/send-code", params={"phone": "13800138000"})
    assert response.status_code == 500  # 你的接口应该返回错误

    app.dependency_overrides.clear()
```

### 3.3 `side_effect` 的多种用法

```python
# 1. 抛异常
mock.side_effect = ValueError("参数错误")

# 2. 每次调用返回不同值（第一次成功，第二次失败，第三次成功...）
mock.side_effect = [
    {"status": "ok"},
    Exception("超时"),
    {"status": "ok"},
]

# 3. 根据输入动态返回
def dynamic_response(*args, **kwargs):
    phone = args[0]
    if phone == "13800138000":
        return {"status": "ok"}
    else:
        raise ValueError("无效手机号")

mock.side_effect = dynamic_response
```

### 3.4 用 `patch` 临时替换

当要替换的不是依赖注入的函数，而是模块内部直接调用的函数时，用 `patch`：

```python
from unittest.mock import patch

def test_payment():
    """测试支付接口——绝不真的扣钱"""
    with patch("services.payment.call_alipay") as mock_pay:
        #             ↑ 模块路径.函数名
        mock_pay.return_value = {"trade_no": "fake-2024", "status": "SUCCESS"}

        client = TestClient(app)
        response = client.post("/pay", json={"order_id": 1, "amount": 9.9})

        assert response.status_code == 200
        mock_pay.assert_called_once()

    # with 退出后，call_alipay 自动恢复原样
```

## 4. 测试认证接口（不真实登录）

测试需要管理员权限的端点时，每次都走登录流程很繁琐。可以直接覆盖 `get_current_user` 依赖：

```python
# conftest.py 或 test文件
from unittest.mock import MagicMock
from auth import get_current_user
from models import User

@pytest.fixture
def admin_client(client):
    """返回一个"以 admin 身份登录"的 TestClient"""
    fake_admin = User(
        id=1,
        username="admin",
        email="admin@test.com",
        role="admin",
        is_active=True,
    )

    def override_get_current_user():
        return fake_admin

    app.dependency_overrides[get_current_user] = override_get_current_user
    yield client
    app.dependency_overrides.clear()

@pytest.fixture
def normal_user_client(client):
    """以普通用户身份登录"""
    fake_user = User(
        id=2,
        username="user",
        email="user@test.com",
        role="user",
        is_active=True,
    )

    def override_get_current_user():
        return fake_user

    app.dependency_overrides[get_current_user] = override_get_current_user
    yield client
    app.dependency_overrides.clear()
```

测试时就非常简洁：

```python
def test_admin_can_see_all_users(admin_client):
    """管理员能看到所有用户"""
    response = admin_client.get("/admin/users")
    assert response.status_code == 200

def test_normal_user_blocked_from_admin(normal_user_client):
    """普通用户访问管理员接口 → 403"""
    response = normal_user_client.get("/admin/users")
    assert response.status_code == 403
```

### 更灵活的方式：参数化的认证 fixture

```python
import pytest

@pytest.fixture
def auth_client(client):
    """工厂 fixture：按需生成任意角色的客户端"""
    def _make_client(role="user", user_id=1, is_active=True):
        fake_user = User(
            id=user_id,
            username=f"{role}_test",
            email=f"{role}@test.com",
            role=role,
            is_active=is_active,
        )

        def override():
            return fake_user

        app.dependency_overrides[get_current_user] = override
        return client
    yield _make_client
    app.dependency_overrides.clear()

def test_multiple_roles(auth_client):
    """一个 fixture 覆盖多种角色"""
    # admin
    r = auth_client(role="admin").get("/admin/users")
    assert r.status_code == 200

    # editor
    r = auth_client(role="editor").post("/posts", json={"title": "t", "content": "c"})
    assert r.status_code == 200

    # normal user
    r = auth_client(role="user").post("/posts", json={"title": "t", "content": "c"})
    assert r.status_code == 403  # user 不能发文章
```

## 5. Mock 外部 HTTP 调用

当代码使用 `httpx` 或 `requests` 直接调用外部 API 时（没有经过依赖注入），可以用专门的 Mock 库。

### 5.1 使用 `responses` 库 Mock requests

```bash
pip install responses
```

```python
import responses

@responses.activate
def test_fetch_weather():
    # Mock 外部 API 的响应
    responses.add(
        responses.GET,
        "https://api.weather.com/beijing",
        json={"temp": 25, "condition": "晴"},
        status=200,
    )

    client = TestClient(app)
    response = client.get("/weather?city=beijing")
    assert response.json()["temp"] == 25
```

### 5.2 使用 `httpx.MockTransport` Mock httpx

如果你的应用使用 `httpx.AsyncClient`（异步），用内置的 Mock Transport：

```python
import httpx
from unittest.mock import MagicMock

# FastAPI TestClient 本身就基于 httpx
def test_call_external_async():
    # 创建一个假的 transport
    async def fake_handler(request):
        return httpx.Response(200, json={"result": "mocked"})

    transport = httpx.MockTransport(fake_handler)
    mock_client = httpx.Client(transport=transport)

    # 把 mock_client 注入到你的依赖中...
```

## 6. 完整示例：测试一个完整的业务流程

我们以"用户注册 → 验证码 → 注册成功"为例子，演示如何把依赖覆盖和 Mock 结合起来。

### 6.1 业务代码

```python
# services/sms.py
import httpx

async def send_sms(phone: str, code: str) -> dict:
    """真人真事：调用第三方短信平台"""
    async with httpx.AsyncClient() as client:
        resp = await client.post(
            "https://sms.example.com/send",
            json={"phone": phone, "message": f"验证码：{code}"},
            timeout=5,
        )
        return resp.json()

# routers/auth.py
from fastapi import APIRouter, Depends, HTTPException
from services.sms import send_sms
from database import get_db
from models import User
from schemas import UserCreate

router = APIRouter()

# 内存中存验证码（生产环境用 Redis）
CODE_STORE: dict[str, str] = {}

@router.post("/send-code")
async def send_code(phone: str, sms=Depends(send_sms)):
    import random
    code = str(random.randint(100000, 999999))
    result = await sms(phone, code)
    CODE_STORE[phone] = code  # 存起来等注册时比对
    return {"message": "验证码已发送"}

@router.post("/register")
async def register(data: UserCreate, db=Depends(get_db)):
    # 验证验证码
    saved_code = CODE_STORE.get(data.phone)
    if not saved_code or saved_code != data.verification_code:
        raise HTTPException(400, "验证码错误")

    # 创建用户
    user = User(phone=data.phone, password=get_password_hash(data.password))
    db.add(user)
    db.commit()
    return {"id": user.id, "phone": user.phone}
```

### 6.2 测试代码

```python
# tests/test_auth_flow.py
import pytest
from unittest.mock import MagicMock, AsyncMock
from fastapi.testclient import TestClient
from main import app
from services.sms import send_sms
from database import get_db

# ── conftest fixture ──
@pytest.fixture
def client(db_session):
    """测试客户端，覆盖数据库和短信"""
    # 覆盖数据库
    def override_db():
        yield db_session
    app.dependency_overrides[get_db] = override_db

    # 覆盖短信服务
    mock_sms = AsyncMock()  # ← async 函数用 AsyncMock！
    mock_sms.return_value = {"status": "ok", "msg_id": "test-001"}
    app.dependency_overrides[send_sms] = mock_sms

    with TestClient(app) as c:
        yield c
    app.dependency_overrides.clear()


# ── 测试用例 ──

def test_full_registration_flow(client):
    """完整的注册流程测试"""

    # 1. 发送验证码
    r = client.post("/send-code", params={"phone": "13900001111"})
    assert r.status_code == 200

    # 2. 用错的验证码注册 → 失败
    r = client.post("/register", json={
        "phone": "13900001111",
        "password": "pass123456",
        "verification_code": "000000"  # 错的
    })
    assert r.status_code == 400

    # 3. 用对的验证码注册 → 成功
    # （从内存中取正确的验证码——这里简化了，实际可以从 Mock 的 call_args 中取）
    from routers.auth import CODE_STORE
    correct_code = CODE_STORE["13900001111"]

    r = client.post("/register", json={
        "phone": "13900001111",
        "password": "pass123456",
        "verification_code": correct_code
    })
    assert r.status_code == 200
    assert r.json()["phone"] == "13900001111"
```

### 6.3 测试外部服务超时的降级

```python
import pytest

def test_sms_timeout_handling(client):
    """短信平台超时 → 接口应返回友好错误"""
    mock_sms = AsyncMock()
    mock_sms.side_effect = TimeoutError("连接超时")

    app.dependency_overrides[send_sms] = mock_sms

    r = client.post("/send-code", params={"phone": "13900001111"})
    assert r.status_code == 500
    assert "超时" in r.json().get("detail", "")

    app.dependency_overrides.clear()
```

## 7. 依赖覆盖的最佳实践

```
依赖覆盖三原则：
  1. 覆盖粒度：覆盖"叶子节点"依赖，不覆盖业务逻辑
  2. 覆盖范围：每个测试只覆盖必要的依赖，不要"偷懒全 Mock"
  3. 覆盖清理：用 fixture 或 try-finally 确保结束后 clear()
```

### 善用 fixture 层级

```python
# conftest.py  —— 分层设计
@pytest.fixture(scope="session")       # 整个测试会话只做一次
def engine():
    """数据库引擎——最外层，创建一次"""
    ...

@pytest.fixture(scope="function")      # 每个测试函数重新创建
def db_session(engine):
    """数据库会话——保证测试隔离"""
    ...

@pytest.fixture(scope="function")
def client(db_session):
    """TestClient——注入 db_session"""
    ...

@pytest.fixture
def admin_client(client):
    """管理员身份的 client——在 client 基础上叠加认证"""
    ...
```

### 常见错误

| 错误 | 正确做法 |
|------|---------|
| 覆盖后忘记 `clear()` | 用 fixture 的 yield + finally 自动清理 |
| Mock 了同步函数却用 `AsyncMock` | 同步用 `MagicMock`，异步用 `AsyncMock` |
| Mock 了但不验证调用 | 用 `assert_called_once()` 等确认 Mock 被正确使用 |
| 过度 Mock——连自己写的工具函数都 Mock | 只 Mock 外部边界（网络、文件系统、数据库） |
| Mock 返回值不真实 | 返回与实际 API 结构一致的假数据，否则测试没意义 |

## 本章小结

```
依赖覆盖的核心价值：
  fast → 不调外部服务，测试毫秒级
  free → 不发短信、不扣钱、不消耗 API 额度
  reliable → 不受网络、第三方服务状态影响
  controllable → 你能模拟任何边界情况（超时、失败、异常返回）
```

| 覆盖什么 | 用什么 |
|---------|--------|
| 数据库（get_db） | 覆盖为 SQLite :memory: |
| 当前用户（get_current_user） | 覆盖为固定的假 User 对象 |
| 短信/邮件/支付 | `MagicMock` 或 `AsyncMock` |
| 第三方 HTTP API | `responses` / `httpx.MockTransport` |
| 文件系统操作 | `tempfile` / `pyfakefs` |
