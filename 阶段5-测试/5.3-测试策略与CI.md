# 5.3 测试策略与 CI

> 测试不是越多越好——学会分层、度量覆盖率、把测试嵌入 CI 流水线，让质量保障成为团队的习惯而非负担。

---

## 1. 测试金字塔：该写多少测试？

测试不是"全都要写"，也不是"随便写几个就行"。经典的测试金字塔告诉你每种测试该占多大比例：

```
        ╱  E2E  ╲           ← 最少：验证核心用户流程是否通
       ╱─────────╲
      ╱  集成测试  ╲          ← 较少：验证模块之间是否协作正常
     ╱───────────────╲
    ╱    单元测试      ╲       ← 最多：验证每个函数/类是否正确
   ╱─────────────────────╲
```

### 三种测试的对比

| | 单元测试 | 集成测试 | E2E 测试 |
|------|---------|---------|---------|
| **测什么** | 单个函数/类 | 多个模块协作 | 完整用户流程 |
| **速度快** | 毫秒级 | 百毫秒级 | 秒级 |
| **可维护性** | 高 | 中 | 低（UI 动一下就坏） |
| **发现问题** | 逻辑错误 | 接口契约错误 | 系统级问题 |
| **Mock 程度** | 全 Mock 外部依赖 | 部分 Mock | 尽量不 Mock |
| **推荐比例** | ~60% | ~30% | ~10% |

### 实例：用户注册该写哪些测试？

```
单元测试（多写）：
  ✅ 密码哈希函数：输入明文，输出哈希，反转失败
  ✅ 密码验证函数：正确密码 True，错误密码 False
  ✅ 验证码生成：长度 6 位，全是数字
  ✅ Pydantic 模型验证：空用户名 → ValidationError

集成测试（适量）：
  ✅ POST /register → 数据库写入 → 返回 200
  ✅ 重复用户名注册 → 返回 409
  ✅ 注册 → 登录 → 拿到 Token → 访问 /me
  ✅ 未登录访问 /me → 401

E2E 测试（少而精）：
  ✅ 打开注册页 → 填表 → 提交 → 跳转登录页 → 登录 → 看到首页
```

## 2. 代码覆盖率：度量你的测试质量

### 2.1 安装和运行

```bash
pip install pytest-cov coverage

# 运行测试并生成覆盖率报告
pytest tests/ --cov=app --cov-report=term-missing

# 输出示例：
# Name                     Stmts   Miss  Cover   Missing
# --------------------------------------------------------------
# app/main.py                 25      0   100%
# app/database.py             15      2    87%   18, 22
# app/models.py               30      0   100%
# app/routers/users.py        45     12    73%   50-62, 80-85
# app/routers/posts.py        38     20    47%   42-78
# app/auth.py                 28      0   100%
# --------------------------------------------------------------
# TOTAL                      181     34    81%
```

### 2.2 覆盖率目标

| 项目阶段 | 建议覆盖率 | 说明 |
|---------|-----------|------|
| 快速原型 / Hackathon | ≥ 50% | 核心流程有测就行 |
| 早期产品 | ≥ 70% | 关键业务路径覆盖 |
| 成长期 | ≥ 80% | 大部分逻辑有保护 |
| 成熟期 / 严格项目 | ≥ 90% | 几乎全部代码有测试 |

> ⚠️ **覆盖率不是目的，发现 Bug 才是。** 100% 覆盖率不代表没 Bug（可能只覆盖了执行路径，没覆盖边界条件）。

### 2.3 生成 HTML 报告

```bash
pytest tests/ --cov=app --cov-report=html

# 打开 htmlcov/index.html，可视化看到每一行是否被测试执行过
# 绿色 = 执行过，红色 = 没执行
```

### 2.4 配置覆盖率的忽略

有些代码不需要测试覆盖，该声明跳过：

```python
# 在代码中标记：这行不要算覆盖率
if TYPE_CHECKING:  # pragma: no cover
    from some_module import SomeType

# 配置文件中排除
# .coveragerc 或 pyproject.toml

# pyproject.toml
[tool.coverage.report]
exclude_lines = [
    "pragma: no cover",
    "if __name__ == .__main__.:",
    "raise NotImplementedError",
    "if TYPE_CHECKING:",
]
```

## 3. CI/CD：让测试自动运行

### 3.1 为什么需要 CI？

```
你改了一行代码，本地跑了测试 → 通过 → 提交 → 合并到 main

3 天后，另一同事也改了同一块代码 → 合并了 → 线上炸了

问题在哪？没有人把那两段改动合在一起跑测试。
```

CI（Continuous Integration，持续集成）做的事就一句话：**每次提交代码，自动跑一遍全套测试。**

### 3.2 GitHub Actions 配置

在项目根目录创建 `.github/workflows/test.yml`：

```yaml
# .github/workflows/test.yml
name: Test

# ── 触发条件 ──
on:
  push:
    branches: [main, develop]     # 推送到这些分支时触发
  pull_request:
    branches: [main]               # 向 main 发 PR 时触发

jobs:
  test:
    runs-on: ubuntu-latest         # 运行环境
    strategy:
      matrix:
        python-version: ["3.10", "3.11", "3.12"]  # 多版本矩阵

    steps:
    # 1. 检出代码
    - name: Checkout
      uses: actions/checkout@v4

    # 2. 设置 Python
    - name: Setup Python ${{ matrix.python-version }}
      uses: actions/setup-python@v5
      with:
        python-version: ${{ matrix.python-version }}

    # 3. 安装依赖
    - name: Install dependencies
      run: |
        python -m pip install --upgrade pip
        pip install -r requirements.txt
        pip install pytest pytest-cov

    # 4. 运行测试
    - name: Run tests
      run: |
        pytest tests/ -v --cov=app --cov-report=xml --cov-report=term

    # 5. 上传覆盖率到 Codecov（可选）
    - name: Upload coverage
      uses: codecov/codecov-action@v4
      with:
        file: ./coverage.xml
        fail_ci_if_error: false
```

### 3.3 完整的 CI 流水线

一个真实项目通常不止跑测试，还有代码检查和构建镜像：

```yaml
# .github/workflows/ci.yml
name: CI Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  # ── 1. 代码风格检查 ──
  lint:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
    - uses: actions/setup-python@v5
      with:
        python-version: "3.11"
    - run: pip install ruff
    - name: Lint with ruff
      run: ruff check .

  # ── 2. 类型检查（可选）──
  typecheck:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
    - uses: actions/setup-python@v5
      with:
        python-version: "3.11"
    - run: pip install mypy
    - name: Type check
      run: mypy app/

  # ── 3. 测试（核心）──
  test:
    needs: [lint]                     # 先 lint 通过才跑测试
    runs-on: ubuntu-latest
    strategy:
      matrix:
        python-version: ["3.10", "3.11", "3.12"]

    services:
      # 如果你的测试需要 MySQL（非 SQLite 内存库时）
      mysql:
        image: mysql:8.0
        env:
          MYSQL_ROOT_PASSWORD: test_password
          MYSQL_DATABASE: test_db
        ports:
          - 3306:3306
        options: >-
          --health-cmd="mysqladmin ping"
          --health-interval=10s
          --health-timeout=5s
          --health-retries=3

    steps:
    - uses: actions/checkout@v4
    - uses: actions/setup-python@v5
      with:
        python-version: ${{ matrix.python-version }}
    - run: pip install -r requirements.txt pytest pytest-cov
    - name: Run tests
      env:
        DATABASE_URL: mysql+pymysql://root:test_password@127.0.0.1:3306/test_db
      run: pytest tests/ -v --cov=app --cov-report=xml

  # ── 4. 构建 Docker 镜像（如有 Dockerfile）──
  build:
    needs: [test]
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'   # 只在 main 分支构建
    steps:
    - uses: actions/checkout@v4
    - name: Build image
      run: docker build -t myapp:${{ github.sha }} .
```

## 4. 常见测试模式

### 4.1 Arrange-Act-Assert（AAA 模式）

每个测试都应该清晰地分为三步：

```python
def test_update_user_email(db, client):
    # ── Arrange（准备）──
    user = User(username="test", email="old@x.com")
    db.add(user)
    db.commit()

    # ── Act（执行）──
    response = client.patch(f"/users/{user.id}", json={"email": "new@x.com"})

    # ── Assert（断言）──
    assert response.status_code == 200
    assert response.json()["email"] == "new@x.com"
```

### 4.2 Given-When-Then（BDD 风格）

```python
def test_admin_can_disable_user(admin_client, db):
    # Given: 有一个激活的用户
    user = User(username="bad_user", is_active=True)
    db.add(user)
    db.commit()

    # When: 管理员禁用这个用户
    response = admin_client.patch(f"/admin/users/{user.id}/toggle-active")

    # Then: 用户变为停用状态
    assert response.status_code == 200
    db.refresh(user)
    assert user.is_active is False
```

### 4.3 边界值测试

```python
class TestRegistrationBoundary:
    """专门测边界的测试类"""

    def test_username_exactly_min_length(self, client):
        """用户名刚好 3 字符 → 通过"""
        r = client.post("/register", json={
            "username": "abc", "email": "a@b.com", "password": "12345678"
        })
        assert r.status_code == 200

    def test_username_one_less_than_min(self, client):
        """用户名 2 字符 → 被拒绝"""
        r = client.post("/register", json={
            "username": "ab", "email": "a@b.com", "password": "12345678"
        })
        assert r.status_code == 422

    def test_password_exactly_min_length(self, client):
        """密码刚好 6 字符 → 通过"""
        r = client.post("/register", json={
            "username": "test", "email": "a@b.com", "password": "123456"
        })
        assert r.status_code == 200

    def test_username_max_length(self, client):
        """用户名 50 字符 → 通过"""
        r = client.post("/register", json={
            "username": "a" * 50, "email": "a@b.com", "password": "12345678"
        })
        assert r.status_code == 200

    def test_username_over_max_length(self, client):
        """用户名 51 字符 → 被拒绝"""
        r = client.post("/register", json={
            "username": "a" * 51, "email": "a@b.com", "password": "12345678"
        })
        assert r.status_code == 422
```

## 5. 测试文档与评审

### 5.1 测试即文档

好的测试本身就是最好的文档——它清楚地展示了一个 API 接受什么输入、返回什么输出：

```python
def test_transfer_money_success():
    """
    ❌ 差的测试名：test_api_1
    ✅ 好的测试名：test_transfer_money_success
    ✅ 更好的：用 docstring 一句话说明场景

    转账成功：余额 100 的用户转 50 给另一个用户
    期望：付款方余额 50，收款方余额 +50，返回交易 ID
    """
    ...

def test_transfer_insufficient_balance():
    """
    转账失败：余额 100 的用户尝试转 200
    期望：返回 422，余额不变，不产生交易记录
    """
    ...
```

### 5.2 Code Review 中的测试检查清单

评审代码时，同时评审测试：

```
□ 新功能有对应的测试吗？
□ 测试覆盖了正常路径和错误路径吗？
□ 测试覆盖了边界值吗？
□ 测试的 Mock 合理吗（没有过度 Mock）？
□ 测试名称清晰表达了测试意图吗？
□ 测试能独立运行吗（不依赖执行顺序）？
□ 有没有"假的测试"——只跑代码不真断言？
□ 测试运行速度合理吗？
```

## 6. 测试运行速度优化

测试慢 → 没人愿意跑 → 写了白写。保持测试在 30 秒内是可行的目标。

```python
# ❌ 每次测试都重新创建表和引擎
@pytest.fixture(scope="function")  # function 级别
def db():
    engine = create_engine("sqlite:///test.db")
    Base.metadata.create_all(engine)  # 每个测试都建表 → 慢
    ...

# ✅ 引擎创建一次，事务回滚代替删表重建
@pytest.fixture(scope="session")
def engine():
    """整个测试会话只创建一次引擎"""
    return create_engine("sqlite:///:memory:", connect_args={"check_same_thread": False})

@pytest.fixture(scope="session")
def tables(engine):
    """整个测试会话只建表一次"""
    Base.metadata.create_all(engine)
    yield
    Base.metadata.drop_all(engine)

@pytest.fixture
def db(engine, tables):
    """每个测试用事务包裹，结束后回滚——超快！"""
    connection = engine.connect()
    transaction = connection.begin()
    session = Session(bind=connection)

    yield session

    transaction.rollback()  # ← 回滚而非重建，快 10 倍
    session.close()
    connection.close()
```

### 速度对比

| 方案 | 100 个测试耗时 |
|------|-------------|
| 每个测试建表 + 删表 | ~120 秒 |
| 共享引擎 + 事务回滚 | ~8 秒 |
| 共享引擎 + 事务回滚 + pytest-xdist | ~3 秒（4 核并行） |

```bash
# 安装并行测试插件
pip install pytest-xdist

# 用 4 个进程并行跑
pytest tests/ -n auto
```

## 7. 测试配置管理

用 `pytest.ini` 或用 `pyproject.toml` 统一管理测试配置：

```toml
# pyproject.toml
[tool.pytest.ini_options]
testpaths = ["tests"]
python_files = "test_*.py"
python_functions = "test_*"
python_classes = "Test*"
addopts = [
    "-v",
    "--strict-markers",
    "--tb=short",
    "--cov=app",
    "--cov-report=term-missing",
    "--cov-fail-under=80",    # 覆盖率低于 80% 视为失败
]
markers = [
    "slow: 慢速测试",
    "integration: 需要外部服务的集成测试",
]

[tool.coverage.run]
source = ["app"]
omit = ["*/migrations/*", "*/tests/*"]

[tool.coverage.report]
exclude_lines = [
    "pragma: no cover",
    "if __name__ == .__main__.:",
    "raise NotImplementedError",
    "if TYPE_CHECKING:",
]
```

## 本章小结

```
测试策略全景：

  测试金字塔          CI 流水线              覆盖率
  ┌─────────┐       push / PR              ┌──────┐
  │   E2E   │  10%      │                  │ 80%+ │ ← 目标
  │ 集成测试 │  30%      ▼                  └──────┘
  │ 单元测试 │  60%   GitHub Actions
  └─────────┘       ┌───┴───┐
                    │ Lint  │ → ruff check
                    │ Test  │ → pytest --cov
                    │ Build │ → docker build
                    └───────┘
```

| 要点 | 核心原则 |
|------|---------|
| 测试金字塔 | 单元测试多写，E2E 少而精 |
| 覆盖率 | 80%+ 是合格线，但不追求 100% |
| CI | 每次提交自动跑全套测试 + lint |
| 速度 | 用事务回滚代替重建数据库，用 pytest-xdist 并行 |
| 文档 | 测试名即文档，AAA 模式清晰可读 |

## 实战练习

1. 在本项目中创建 `.github/workflows/test.yml`，配置 push 到 main 时自动运行测试
2. 为你的项目运行 `pytest --cov`，找到覆盖率最低的模块，针对性补充测试
3. 将你的测试按"单元测试"和"集成测试"分目录，用 `@pytest.mark.integration` 标记集成测试
4. 用事务回滚方案优化你当前的测试 fixture（从建表/删表改为 begin/rollback）
5. 配置 `pyproject.toml` 中的 pytest 和 coverage 选项
