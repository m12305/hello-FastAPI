# 3.3 Alembic 数据库迁移

> 数据库结构也需要"Git"——用 Alembic 对每次表结构变更做版本管理，告别手动改表、丢数据的噩梦。

---

## 1. 为什么需要数据库迁移？

在开发过程中，数据库表结构一直在变化：今天加个字段，明天下线一个旧列，后天改类型、加索引……

### 没有迁移工具时

```
开发阶段：
  $ sqlite3 app.db "ALTER TABLE users ADD COLUMN bio TEXT;"
  # 在本地跑完...但队友不知道！

上线时：
  # 手动在一堆服务器上执行 SQL
  # 某台服务器忘了跑 → 上线后疯狂报错 500

数据回滚时：
  # 不知道上一个版本的表结构是什么
  # 备份？备份是三天前的...丢了三天的数据
```

### 有了 Alembic

```
本地开发：
  $ alembic revision --autogenerate -m "add bio to users"
  # 自动生成迁移文件 users_add_bio.py

代码提交：
  $ git add migrations/versions/xxx_add_bio.py
  $ git commit -m "migration: add bio to users"

队友拉代码：
  $ git pull
  $ alembic upgrade head  # 一键更新本地数据库

上线：
  $ alembic upgrade head  # 自动执行所有未应用的迁移

回滚：
  $ alembic downgrade -1  # 撤销最后一次迁移
```

> Alembic 之于数据库，就像 Git 之于代码。

## 3. 配置 Alembic 连接到你的数据库

### 步骤 1：修改 `alembic.ini`

```ini
# alembic.ini
# 找到 sqlalchemy.url 这一行，改成你的数据库 URL
sqlalchemy.url = sqlite:///./app.db

# 生产环境更常见的做法是注释掉这行，改用 env.py 动态读取
```

### 步骤 2：修改 `alembic/env.py`（关键！）

```python
"""alembic/env.py — Alembic 环境配置"""
from logging.config import fileConfig
from sqlalchemy import engine_from_config, pool
from alembic import context

# ═══════════ 导入你的模型和 Base ═══════════
# 必须导入！否则 Alembic 不知道有哪些表
import sys
from pathlib import Path

# 把项目根目录加入 Python 路径
sys.path.insert(0, str(Path(__file__).resolve().parent.parent))

from database import Base, DATABASE_URL
from models import User, Post  # ← 导入所有模型，让 Base.metadata 注册它们

# ═══════════ Alembic 标准配置 ═══════════
config = context.config

# 如果用 alembic.ini 的 sqlalchemy.url
config.set_main_option("sqlalchemy.url", DATABASE_URL)

# （可选）从环境变量或 .env 文件读取
# import os
# config.set_main_option("sqlalchemy.url", os.getenv("DATABASE_URL"))

if config.config_file_name is not None:
    fileConfig(config.config_file_name)

# ═══════════ 告诉 Alembic 你的表结构 ═══════════
target_metadata = Base.metadata  # ← 核心：Alembic 对比这个来生成迁移


def run_migrations_offline():
    """离线模式：生成 SQL 文件而非直接执行"""
    url = config.get_main_option("sqlalchemy.url")
    context.configure(
        url=url,
        target_metadata=target_metadata,
        literal_binds=True,
        dialect_opts={"paramstyle": "named"},
    )
    with context.begin_transaction():
        context.run_migrations()


def run_migrations_online():
    """在线模式：直接连接数据库执行"""
    connectable = engine_from_config(
        config.get_section(config.config_ini_section),
        prefix="sqlalchemy.",
        poolclass=pool.NullPool,
    )
    with connectable.connect() as connection:
        context.configure(
            connection=connection,
            target_metadata=target_metadata
        )
        with context.begin_transaction():
            context.run_migrations()


if context.is_offline_mode():
    run_migrations_offline()
else:
    run_migrations_online()
```

## 4. 核心命令

```bash
# 生成迁移（自动对比 models.py ↔ 当前数据库）
alembic revision --autogenerate -m "描述信息"

# 执行迁移（升级到最新版本）
alembic upgrade head

# 回滚一步
alembic downgrade -1

# 回滚到最初（删掉所有表）
alembic downgrade base

# 查看当前版本
alembic current

# 查看迁移历史
alembic history

# 生成 SQL 而不执行（审查用）
alembic upgrade head --sql
```

## 5. 完整工作流演示

### 场景：给 User 表新增 `bio` 字段

**第 1 步：修改模型**

```python
# models.py — 在 User 类中添加
class User(Base):
    # ... 原有字段 ...
    bio: Mapped[str | None] = mapped_column(String(500), default=None)
```

**第 2 步：生成迁移**

```bash
$ alembic revision --autogenerate -m "add bio to users"

# 输出：
# INFO  [alembic.autogenerate.compare] Detected added column 'users.bio'
# Generating /path/to/alembic/versions/abc123_add_bio_to_users.py ... done
```

**第 3 步：检查生成的迁移文件**

```python
"""abc123_add_bio_to_users.py — Alembic 自动生成的迁移（务必检查！）"""
revision = 'abc123_def456'
down_revision = 'xyz789_uvw012'
branch_labels = None
depends_on = None

from alembic import op
import sqlalchemy as sa


def upgrade():
    """从旧版本 → 新版本"""
    op.add_column('users', sa.Column('bio', sa.String(length=500), nullable=True))


def downgrade():
    """从新版本 → 旧版本（回滚）"""
    op.drop_column('users', 'bio')
```

> ⚠️ **重要：** 自动生成的迁移文件必须人工检查！Alembic 不是万能的——比如列重命名它可能识别为"删了一列，加了一列"。

**第 4 步：执行迁移**

```bash
$ alembic upgrade head

# INFO  [alembic.runtime.migration] Running upgrade xyz789 → abc123, add bio to users
```

**第 5 步：如果需要回滚**

```bash
$ alembic downgrade -1
# INFO  [alembic.runtime.migration] Running downgrade abc123 → xyz789
```

## 6. Alembic 不能自动检测的操作

| 操作 | 自动检测？ | 怎么办 |
|------|-----------|--------|
| 新增表 | ✅ 能 | 直接用 |
| 新增列 | ✅ 能 | 直接用 |
| 删除列 | ✅ 能 | 直接用 |
| 修改列类型 | ⚠️ 部分 | 检查生成的代码 |
| **重命名列** | ❌ 不能 | 手动写 `op.alter_column('table', 'old_name', new_column_name='new_name')` |
| **重命名表** | ❌ 不能 | 手动写 `op.rename_table('old_name', 'new_name')` |
| 创建索引/约束 | ⚠️ 部分 | 需手动命名 |
| 数据迁移 | ❌ 不能 | 手动在 `upgrade()` 中写 SQL |

### 手动编写数据迁移

```python
"""手动迁移示例：拆分 full_name 为 first_name + last_name"""
revision = 'def456'
down_revision = 'abc123'

from alembic import op
import sqlalchemy as sa


def upgrade():
    # 1. 加新列
    op.add_column('users', sa.Column('first_name', sa.String(50), nullable=True))
    op.add_column('users', sa.Column('last_name', sa.String(50), nullable=True))

    # 2. 数据迁移（把旧数据拆分到新列）
    conn = op.get_bind()
    conn.execute(sa.text("""
        UPDATE users
        SET first_name = SUBSTR(full_name, 1, INSTR(full_name, ' ') - 1),
            last_name = SUBSTR(full_name, INSTR(full_name, ' ') + 1)
        WHERE full_name LIKE '% %'
    """))

    # 3. 删旧列
    op.drop_column('users', 'full_name')


def downgrade():
    # 回滚：反向操作
    op.add_column('users', sa.Column('full_name', sa.String(100), nullable=True))
    conn = op.get_bind()
    conn.execute(sa.text("""
        UPDATE users
        SET full_name = first_name || ' ' || last_name
    """))
    op.drop_column('users', 'first_name')
    op.drop_column('users', 'last_name')
```

## 7. 生产环境最佳实践

### 原则

```
1️⃣ 永远先检查自动生成的迁移文件
2️⃣ 每个迁移必须写 downgrade()（别留空）
3️⃣ 迁移文件和模型代码一起提交 Git
4️⃣ 不直接在生产数据库手动改表结构
5️⃣ 部署前先在测试数据库上跑迁移
6️⃣ 大表加列/改列用分批策略（避免锁表太久）
```

### CI/CD 集成

```yaml
# .github/workflows/deploy.yml 片段
- name: Run database migrations
  run: |
    alembic upgrade head
  env:
    DATABASE_URL: ${{ secrets.DATABASE_URL }}
```

### 回滚策略

```bash
# ⚠️ 先用 --sql 看看要执行什么
alembic downgrade -1 --sql

# 确认没问题再真正执行
alembic downgrade -1
```

## 8. 常见问题排查

### 问题 1：`alembic revision --autogenerate` 生成了空文件

```bash
# 原因 1：你没有导入模型到 env.py
# 原因 2：模型和数据库已经一致（没有变更）

# 检查：env.py 中确实导入了所有模型
from models import User, Post  # ← 确保这行存在

# 测试：打印看看 Base.metadata 有没有注册表
print(Base.metadata.tables.keys())
# 应该输出: dict_keys(['users', 'posts'])
```

### 问题 2：`alembic upgrade head` 报错"表已存在"

```bash
# 原因：之前用了 Base.metadata.create_all()，然后想接上 Alembic

# 解决：让 Alembic 从当前数据库状态开始
alembic stamp head  # 标记当前状态为最新，不执行任何迁移
```

### 问题 3：多个分支的迁移冲突

```bash
# 每个人的迁移"链"不一样，合并时可能出现两个 head

# 查看有多少个 head
alembic heads

# 合并两个 head
alembic merge head_a head_b -m "merge branches"
```
