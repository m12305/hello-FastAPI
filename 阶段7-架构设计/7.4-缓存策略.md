# 7.4 缓存策略

> 数据库不是唯一的选择——用缓存把热门数据放在"离用户最近的地方"，减少数据库压力、降低响应延迟。

---

## 1. 为什么需要缓存？

### 一个典型的性能瓶颈

```
用户每次刷新页面 → API 查数据库 → 同样的查询执行了 1000 次/秒
                                            │
                                    大部分时间在重复劳动
```

```
没有缓存：                             有缓存：
  用户请求                              用户请求
    │                                     │
    ▼                                     ▼
  API 查数据库（每次 50ms）            先查缓存（1ms）
    │                                     │
    ▼                                  ┌─┴─────────┐
  返回数据                             命中         未命中
    （慢）                              │           │
                                        ▼           ▼
                                    返回缓存    查数据库 → 写入缓存
                                      （快）     （50ms）  （下次就快了）
```

| 指标 | 无缓存 | 有缓存 |
|------|--------|--------|
| 响应时间 | 50ms（每次查 DB） | 1ms（命中时） |
| 数据库压力 | 承受全部请求 | 只承受未命中部分 |
| QPS 上限 | 数据库成为瓶颈 | 缓存扛住绝大部分 |

## 2. HTTP 缓存头（最简单、零成本）

### 2.1 Cache-Control

```python
from fastapi import Response

@app.get("/products")
def list_products(response: Response):
    # 告诉浏览器和 CDN：这个数据 5 分钟内不变
    response.headers["Cache-Control"] = "public, max-age=300"
    return {"products": [...]}

@app.get("/user/profile")
def get_profile(response: Response):
    # 私有数据：只允许浏览器缓存，CDN 不能缓存
    response.headers["Cache-Control"] = "private, max-age=60"
    return {"username": "zhangsan"}
```

| 指令 | 含义 | 适用 |
|------|------|------|
| `public` | 允许 CDN 和代理缓存 | 公共数据（文章列表） |
| `private` | 只允许浏览器缓存 | 用户数据（个人信息） |
| `max-age=300` | 缓存 300 秒 | 可接受一定延迟的数据 |
| `no-cache` | 每次用前验证 | 需要保证时效性 |
| `no-store` | 完全不缓存 | 敏感数据 |

### 2.2 ETag（条件请求）

```python
import hashlib
import json
from fastapi import Request, Response

@app.get("/products/{product_id}")
def get_product(product_id: int, request: Request, response: Response):
    product = db.get(Product, product_id)
    data = {"id": product.id, "name": product.name, "price": product.price}

    # 计算 ETag：数据内容的哈希值
    etag = hashlib.md5(json.dumps(data, sort_keys=True).encode()).hexdigest()

    # 客户端带来的 ETag 和当前一致 → 数据没变 → 304 Not Modified
    if request.headers.get("If-None-Match") == etag:
        response.status_code = 304
        return Response(status_code=304)

    response.headers["ETag"] = etag
    response.headers["Cache-Control"] = "public, max-age=60"
    return data
```

```
客户端首次请求：
  GET /products/1
  ← 200 OK, ETag: "abc123", 数据: {...}

客户端再次请求（数据没变）：
  GET /products/1, If-None-Match: "abc123"
  ← 304 Not Modified（无 body，节省带宽！）
```

## 3. 内存缓存（适合单机、开发阶段）

### 3.1 最简单的 dict 缓存

```python
from functools import lru_cache

# ── 方案 A：LRU 缓存（最近最少使用）──
@lru_cache(maxsize=128)
def get_product_by_id(product_id: int):
    """自动缓存最近 128 次查询的结果"""
    return db.get(Product, product_id)
# 注意：只适用于无参数之外的依赖（如数据库不变）的场景
```

### 3.2 带 TTL 的 dict 缓存

```python
import time
from threading import Lock

class TTLCache:
    """
    带过期时间的本地缓存。

    Args:
        ttl_seconds: 过期时间（秒）
    """
    def __init__(self, ttl_seconds: int = 300):
        self._store: dict[str, tuple[float, any]] = {}
        self._ttl = ttl_seconds
        self._lock = Lock()

    def get(self, key: str):
        with self._lock:
            if key in self._store:
                expires_at, value = self._store[key]
                if time.time() < expires_at:
                    return value
                del self._store[key]  # 过期了，删除
        return None

    def set(self, key: str, value):
        with self._lock:
            self._store[key] = (time.time() + self._ttl, value)

    def delete(self, key: str):
        with self._lock:
            self._store.pop(key, None)


# 使用
cache = TTLCache(ttl_seconds=300)  # 5 分钟过期

@app.get("/products/{product_id}")
def get_product(product_id: int):
    # 先查缓存
    cached = cache.get(f"product:{product_id}")
    if cached:
        return {"source": "cache", "data": cached}

    # 缓存未命中 → 查 DB → 写入缓存
    product = db.get(Product, product_id)
    if not product:
        raise HTTPException(404)
    data = {"id": product.id, "name": product.name}
    cache.set(f"product:{product_id}", data)
    return {"source": "database", "data": data}
```

## 4. Redis 缓存（生产环境标准方案）

### 4.1 安装与连接

```bash
pip install redis
# 需要启动 Redis 服务：docker run -d -p 6379:6379 redis:7
```

```python
# app/core/dependencies.py
import redis
from app.core.config import settings

# 在 lifespan 中创建 Redis 客户端
import redis.asyncio as async_redis  # 异步版本

redis_client: async_redis.Redis | None = None

async def get_redis() -> async_redis.Redis:
    """依赖注入：获取 Redis 客户端"""
    if redis_client is None:
        raise HTTPException(503, "Redis 不可用")
    return redis_client
```

### 4.2 Cache-Aside 模式（最常用）

```
Cache-Aside（旁路缓存）：
  读取: 先查缓存 → 命中返回（快） → 未命中查 DB → 写缓存（下次就快）
  写入: 先写 DB → 再删缓存（或更新缓存） → 下次读取重新加载
```

```python
import json
from redis.asyncio import Redis

@app.get("/products/{product_id}")
async def get_product(product_id: int, redis: Redis = Depends(get_redis)):
    """Cache-Aside 读取模式"""
    # 1. 先查缓存
    cache_key = f"product:{product_id}"
    cached = await redis.get(cache_key)

    if cached:
        return {"source": "cache", "data": json.loads(cached)}

    # 2. 缓存未命中 → 查 DB
    product = db.get(Product, product_id)
    if not product:
        raise HTTPException(404)

    data = {"id": product.id, "name": product.name, "price": product.price}

    # 3. 写入缓存（TTL 5 分钟）
    await redis.setex(cache_key, 300, json.dumps(data, ensure_ascii=False))

    return {"source": "database", "data": data}


@app.patch("/products/{product_id}")
async def update_product(
    product_id: int,
    data: ProductUpdate,
    redis: Redis = Depends(get_redis),
):
    """Cache-Aside 写入模式：先写 DB → 再删缓存"""
    product = db.get(Product, product_id)
    if not product:
        raise HTTPException(404)

    # 更新数据库
    product.name = data.name
    db.commit()

    # 删除缓存（下次读取时重新加载最新数据）
    await redis.delete(f"product:{product_id}")

    return {"message": "已更新"}
```

### 4.3 批量预热缓存

```python
async def warm_up_cache():
    """应用启动时预加载热门数据到缓存"""
    products = db.query(Product).filter(Product.is_popular == True).all()
    for p in products:
        key = f"product:{p.id}"
        value = json.dumps({"id": p.id, "name": p.name, "price": p.price})
        await redis_client.setex(key, 300, value)
    print(f"✅ 缓存预热完成: {len(products)} 个商品")


@asynccontextmanager
async def lifespan(app: FastAPI):
    # ...
    await warm_up_cache()  # 启动后立即预热
    yield
    # ...
```

## 5. 缓存模式对比

| 模式 | 读策略 | 写策略 | 适用场景 |
|------|--------|--------|---------|
| **Cache-Aside** | 先查缓存，未命中查 DB | 先写 DB，再删缓存 | 读多写少（最常用） |
| **Read-Through** | 查缓存，由缓存层负责查 DB | — | 缓存层封装完整 |
| **Write-Through** | — | 同时写缓存和 DB | 需要强一致性 |
| **Write-Behind** | — | 先写缓存，异步写 DB | 写入密集场景 |

### 选择指南

```
读多写少 → Cache-Aside
  （文章列表、商品详情、用户信息）

要求数据一致性 → Write-Through
  （库存扣减、余额查询）

写入密集 → Write-Behind
  （计数器、统计数据）
```

## 6. 缓存失效与雪崩预防

### 6.1 缓存穿透、击穿、雪崩

```
缓存穿透：查不存在的数据 → 每次都穿透到 DB
  解决：布隆过滤器 OR 缓存空值（"不存在"也缓存 30 秒）

缓存击穿：热点数据过期 → 瞬间大量请求打到 DB
  解决：互斥锁（只有一个线程去查 DB）、永不过期+异步刷新

缓存雪崩：大量缓存同时过期 → DB 瞬间承受全部压力
  解决：TTL 加随机值（300±60 秒）、多级缓存
```

### 6.2 TTL 加随机抖动

```python
import random

# ❌ 所有缓存在同一秒过期
await redis.setex("key", 300, value)

# ✅ 加随机抖动：287-313 秒之间过期
ttl = 300 + random.randint(-30, 30)
await redis.setex("key", ttl, value)
```

### 6.3 缓存空值（防穿透）

```python
@app.get("/products/{product_id}")
async def get_product(product_id: int, redis: Redis = Depends(get_redis)):
    cache_key = f"product:{product_id}"
    cached = await redis.get(cache_key)

    if cached is not None:
        if cached == "__NULL__":
            raise HTTPException(404, "商品不存在（缓存）")
        return {"source": "cache", "data": json.loads(cached)}

    product = db.get(Product, product_id)
    if not product:
        # 缓存"不存在"的结果，防止反复穿透
        await redis.setex(cache_key, 30, "__NULL__")  # 空值缓存 30 秒
        raise HTTPException(404)

    data = {"id": product.id, "name": product.name}
    await redis.setex(cache_key, 300, json.dumps(data, ensure_ascii=False))
    return {"source": "database", "data": data}
```

## 本章小结

```
缓存策略梯队：

  第一层：HTTP 缓存头（零成本）
    Cache-Control + ETag → 浏览器和 CDN 缓存

  第二层：本地内存缓存（单机）
    TTLCache / functools.lru_cache → 毫秒级

  第三层：Redis 缓存（分布式）
    Cache-Aside 模式 → 多实例共享、持久化
```

| 要点 | 最佳实践 |
|------|---------|
| **HTTP 缓存** | `Cache-Control: public, max-age=300` |
| **模式** | Cache-Aside（读多写少场景首选） |
| **Key 设计** | `业务:ID` 格式：`product:123`、`user:456` |
| **TTL** | 按数据新鲜度：5 分钟（商品）、1 小时（文章）、1 天（配置） |
| **防雪崩** | TTL 加随机值，错开过期时间 |
| **防穿透** | 缓存空值 OR 布隆过滤器 |
