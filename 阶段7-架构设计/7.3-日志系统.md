# 7.3 日志系统

> `print("bug happened!")` 不是日志——用 Python logging + 结构化日志替代 print，让排查问题有迹可循。

---

## 1. 为什么不能用 print？

```python
# ❌ 你在用 print 调试？
print(f"用户 {username} 登录成功")
print(f"数据库查询耗时: {elapsed}")
print(f"出错了！{e}")
```

| print 的问题 | 日志的解决方案 |
|-------------|---------------|
| 没有时间戳 | `2026-06-14 10:23:45 INFO ...` |
| 没有级别区分 | DEBUG / INFO / WARNING / ERROR / CRITICAL |
| 不能写到文件 | `logging.FileHandler` |
| 不能按级别过滤 | `logger.setLevel(logging.WARNING)` 只看警告+错误 |
| 不能集中收集 | Filebeat → ELK / Loki |
| 生产环境关不掉 | 切到 WARNING 级别，print 照样输出 |

> 💡 一句话：**print 是给你临时调试的，logging 是给系统长期运行的。**

## 2. Python logging 快速上手

### 2.1 基础用法

```python
import logging

# ── 创建一个 Logger ──
logger = logging.getLogger("myapp")     # "myapp" 是 logger 的名字
logger.setLevel(logging.DEBUG)          # 设置最低输出级别

# ── 创建一个 Handler（输出到控制台）──
handler = logging.StreamHandler()
handler.setLevel(logging.INFO)

# ── 设置格式 ──
formatter = logging.Formatter(
    "%(asctime)s | %(levelname)-8s | %(name)s | %(message)s",
    datefmt="%Y-%m-%d %H:%M:%S",
)
handler.setFormatter(formatter)

# ── 组合 ──
logger.addHandler(handler)

# ── 使用 ──
logger.debug("这条不会输出（DEBUG < Handler 的 INFO）")
logger.info("用户登录成功")
logger.warning("磁盘使用率 85%")
logger.error("数据库连接失败")
logger.critical("服务器即将宕机！")
```

输出：
```
2026-06-14 10:23:45 | INFO     | myapp | 用户登录成功
2026-06-14 10:23:45 | WARNING  | myapp | 磁盘使用率 85%
2026-06-14 10:23:46 | ERROR    | myapp | 数据库连接失败
2026-06-14 10:23:46 | CRITICAL | myapp | 服务器即将宕机！
```

### 2.2 日志级别

| 级别 | 数值 | 含义 | 使用场景 |
|------|------|------|---------|
| `DEBUG` | 10 | 调试信息 | 开发中：变量值、函数入参出参 |
| `INFO` | 20 | 一般信息 | 生产：请求日志、业务流程记录 |
| `WARNING` | 30 | 警告 | 磁盘快满、配置项缺失但有默认值 |
| `ERROR` | 40 | 错误 | 功能异常但系统还能跑 |
| `CRITICAL` | 50 | 致命 | 系统即将崩溃，需要立即处理 |

## 3. FastAPI 集成日志

### 3.1 统一日志配置

```python
# app/core/logging_config.py
import logging
import sys

def setup_logging(log_level: str = "INFO"):
    """
    统一日志配置 —— 在 app 启动时调用一次。

    特点：
      - 所有 logger 统一格式
      - 控制台输出 + 文件输出
      - 第三方库（uvicorn, sqlalchemy）日志级别独立控制
    """
    # ── 根 logger ──
    root_logger = logging.getLogger()
    root_logger.setLevel(getattr(logging, log_level.upper()))

    # ── 格式 ──
    formatter = logging.Formatter(
        fmt="%(asctime)s | %(levelname)-8s | %(name)-20s | %(message)s",
        datefmt="%Y-%m-%d %H:%M:%S",
    )

    # ── 控制台 Handler ──
    console = logging.StreamHandler(sys.stdout)
    console.setFormatter(formatter)
    root_logger.addHandler(console)

    # ── 文件 Handler（生产环境）──
    file_handler = logging.FileHandler("app.log", encoding="utf-8")
    file_handler.setFormatter(formatter)
    root_logger.addHandler(file_handler)

    # ── 控制第三方库的日志级别 ──
    logging.getLogger("uvicorn.access").setLevel(logging.WARNING)
    logging.getLogger("sqlalchemy.engine").setLevel(logging.WARNING)
    # 想看 SQL 日志时改为 INFO

    root_logger.info(f"日志系统已初始化 (级别: {log_level})")
```

### 3.2 在 lifespan 中初始化

```python
# app/main.py
from contextlib import asynccontextmanager
from app.core.logging_config import setup_logging
from app.core.config import settings

@asynccontextmanager
async def lifespan(app: FastAPI):
    # 启动时初始化日志
    setup_logging(settings.LOG_LEVEL)
    yield

app = FastAPI(lifespan=lifespan)
```

### 3.3 请求日志中间件

```python
import time
import logging
from fastapi import Request

logger = logging.getLogger("app.access")

@app.middleware("http")
async def log_requests(request: Request, call_next):
    """记录每个请求的 method、path、status、耗时"""
    start = time.time()
    response = await call_next(request)
    elapsed = (time.time() - start) * 1000

    logger.info(
        f"{request.method} {request.url.path} → {response.status_code} ({elapsed:.2f}ms)",
        extra={
            "method": request.method,
            "path": request.url.path,
            "status_code": response.status_code,
            "elapsed_ms": round(elapsed, 2),
            "client_ip": request.client.host if request.client else "unknown",
        },
    )
    return response
```

## 4. 结构化日志

### 4.1 为什么需要结构化日志？

```
# ❌ 传统文本日志 —— 人读还行，机器没法分析
2026-06-14 10:23:45 | INFO | app | 用户 zhangsan (id=42) 登录成功，IP: 1.2.3.4

# ✅ 结构化日志 —— 人可读，机器也能自动分析
{"time": "2026-06-14T10:23:45", "level": "INFO", "logger": "app",
 "event": "user_login",
 "username": "zhangsan", "user_id": 42, "ip": "1.2.3.4"}
```

| 文本日志 | 结构化日志 |
|---------|-----------|
| 搜索要靠 grep 猜测 | `user_id=42 AND event=user_login` 精确查询 |
| 不能做聚合 | `COUNT(event) GROUP BY event` |
| 出问题大海捞针 | "过去 1 小时内 login 失败的 TOP 10 用户" |

### 4.2 用 structlog 实现

```bash
pip install structlog
```

```python
# app/core/logging_config.py
import structlog
import logging

def setup_structured_logging():
    """配置结构化日志"""
    structlog.configure(
        processors=[
            structlog.stdlib.filter_by_level,          # 按级别过滤
            structlog.stdlib.add_logger_name,          # 添加 logger 名
            structlog.stdlib.add_log_level,            # 添加级别
            structlog.stdlib.PositionalArgumentsFormatter(),
            structlog.processors.TimeStamper(fmt="iso"),  # 添加时间戳
            structlog.processors.StackInfoRenderer(),
            structlog.processors.format_exc_info,      # 格式化异常
            structlog.processors.UnicodeDecoder(),
            structlog.dev.ConsoleRenderer(),            # 开发环境：彩色输出
            # structlog.processors.JSONRenderer(),     # 生产环境：JSON 输出
        ],
        context_class=dict,
        logger_factory=structlog.stdlib.LoggerFactory(),
        wrapper_class=structlog.stdlib.BoundLogger,
        cache_logger_on_first_use=True,
    )

# 使用
logger = structlog.get_logger()

# ── 绑定上下文 ──
log = logger.bind(user_id=42, request_id="req-abc123")
log.info("user_login", username="zhangsan", ip="1.2.3.4")
# 输出: 2026-06-14T10:23:45 [info] user_login  user_id=42 request_id=req-abc123 username=zhangsan ip=1.2.3.4
```

### 4.3 为每个请求绑定 request_id

```python
# app/middleware/logging.py
import uuid
import structlog
from fastapi import Request

@app.middleware("http")
async def add_request_context(request: Request, call_next):
    """为每个请求生成唯一 ID，绑定到日志上下文"""
    request_id = str(uuid.uuid4())[:8]

    # 将 request_id 绑定到 structlog 的上下文
    structlog.contextvars.bind_contextvars(request_id=request_id)

    # 也可以存到 request.state 中，方便后续使用
    request.state.request_id = request_id

    logger = structlog.get_logger()
    logger.info("request_started", method=request.method, path=request.url.path)

    response = await call_next(request)

    logger.info("request_finished", status_code=response.status_code)
    response.headers["X-Request-ID"] = request_id

    # 清理上下文
    structlog.contextvars.clear_contextvars()
    return response
```

## 6. 日志最佳实践

```
DO ✅:
  1. 模块名作为 logger 名: logging.getLogger(__name__)
  2. 异常用 logger.exception(): 自动附带堆栈跟踪
  3. 敏感信息脱敏: 密码、Token、身份证号不记日志
  4. 关键节点记录: 登录、支付、权限变更
  5. 日志分级正确: DEBUG 不泛滥，ERROR 不忽略

DON'T ❌:
  1. 不要用 print()
  2. 不要在循环中记大量 DEBUG 日志
  3. 不要在日志中写密码/Token/身份证
  4. 不要用字符串拼接: 用 %s 或 f-string 之外的方式（延迟求值）
```

## 本章小结

```
日志系统全景：

  Python logging              structlog                  ELK/Loki
  (内置、零依赖)           (结构化、上下文)          (集中收集、可视化)
        │                       │                         │
  开发阶段使用             团队/生产使用              大规模生产使用
```

| 要点 | 最佳实践 |
|------|---------|
| **库选择** | 小项目用 `logging`，进阶用 `structlog` |
| **格式** | 开发环境彩色文本，生产环境 JSON |
| **中间件** | 记录每个请求的 method / path / status / 耗时 |
| **request_id** | 每个请求一个唯一 ID，关联所有日志 |
| **日志收集** | 单机文件 → Loki → ELK，逐步升级 |
